## 1. 目标与结论
- 该需求对应 quiz 里的“单词熟练度统计”专题报表，核心不是单独的内容管理页，而是 **围绕单词卡复习状态做学习成效观察**：看当前用户有多少单词、多少仍活跃、多少今天应复习、多少已进入稳定掌握阶段，以及最近复习打分和近 7 天复习活跃度。
- 在 nquiz（Next.js 重构版）里，不建议机械复刻旧的 `statistics-center` 入口。更合理的是把它作为 **Vocabulary 域下的学习洞察页** 迁移，例如直接挂到 `vocabulary/proficiency` 一类叶子菜单，而不是继续保留老的统计中心目录壳。
- 迁移时应保留旧页面的 4 个核心指标 + 3 张核心图，但要补上更清晰的信息架构、加载/空态/错误态、与“去复习/看单词列表”的业务闭环，以及更符合 nquiz 技术栈的组件与状态管理实现。

## 3. 当前真实功能边界
### 页面当前展示内容
旧页面实际由以下内容组成：
1. **4 个 KPI 卡片**
   - 单词总数 `totalWords`
   - 活跃单词 `activeWords`
   - 今日待复习 `dueTodayWords`
   - 已熟练 `masteredWords`
2. **2 个均值指标**
   - 平均连续记对次数 `averageRepetition`
   - 平均简易度因子 `averageEasinessFactor`
3. **3 张图**
   - 熟练度分层分布 `proficiencyDistribution`
   - 复习评分分布 `reviewScoreDistribution`
   - 近七天复习趋势 `reviewCountByLastSevenDays`

### 后端真实统计口径
`VocabularyProficiencyStatisticsServiceImpl` 里已经把口径写死了：
1. **数据范围仅限当前登录用户**：所有 SQL 都按 `create_user = 当前用户` 过滤，不是团队/班级视角。
2. **总数/活跃/归档**：来自 `vocabulary_card`。
3. **已熟练**：未归档且 `repetition >= 6`。
4. **今日待复习**：未归档且 `next_review_date <= now`。
5. **平均连续记对次数 / 平均简易度因子**：仅统计未归档卡片。
6. **熟练度分层**：
   - 0 次：新词
   - 1-2 次：入门
   - 3-5 次：进阶
   - 6+ 次：熟练
7. **复习评分分布**：来自 `review_log`，分数桶为 0~5 分。
8. **近七天趋势**：统计最近 7 天每天的复习次数。

### 与单词学习主链路的关系
这不是独立报表，而是 Vocabulary 学习闭环的观察面板：
- 列表页 `Vocabulary/index.tsx` 明确展示 `repetition`、`nextReviewDate`、`easinessFactor`；
- 复习页 `Vocabulary/Review.tsx` 通过 `getDueToday()` 拉取今日待复习卡，再通过 `reviewVocabulary()` 提交评分；
- 因此“熟练度统计”本质上是对 **单词卡 + 复习行为** 的二次聚合展示。

## 4. 现状问题与迁移判断
### 旧实现存在的问题
1. **UI 是报表孤岛**：旧页面只有读数据，没有直接连接“去复习”“查看对应单词”的操作闭环。
2. **状态管理较原始**：页面用 `useEffect + useState` 手动拉数，缺少缓存、重试、统一错误态。
3. **图表实现偏命令式**：直接 `echarts.init/ref/dispose`，在 React 重构版里维护成本偏高。
4. **没有筛选/钻取能力**：只能看当前用户全量快照，不能按归档、标签、创建时间、掌握阶段下钻。
5. **历史入口已经被下线**：旧系统连 `statistics-center` 菜单都已删掉，说明“作为独立统计中心存在”这件事本身并不是必须继承的产品约束。
6. **评分分布与前端打分按钮存在语义收窄**：当前复习页按钮只提供 0 / 3 / 5 三档，但统计接口仍按 0~5 分桶，这会导致中间分值常年为空桶或仅由历史数据填充，nquiz 里要决定是保留 0~5 完整分布，还是按现行交互收敛为更贴近用户心智的分档。

### 对 nquiz 的迁移判断
- **应保留**：核心学习指标、熟练度分层、评分分布、近 7 天趋势。
- **应优化**：页面归属、信息架构、交互闭环、状态管理、加载/空态、图表实现方式。
- **可以重组或延后**：复杂筛选、导出、跨用户/班级对比、标签/学科维度分析。这些旧接口并未提供，不应在首版迁移中硬塞。

## 5. nquiz / Next.js 推荐实现方案
### 5.1 页面定位
建议不要恢复 `statistics-center` 总入口，而是直接把页面放到单词域，例如：
- `app/(dashboard)/vocabulary/proficiency/page.tsx`

这样更符合真实业务语义：这是 vocabulary 模块的学习洞察页，而不是独立的全局 BI 系统。

### 5.2 信息架构
建议分成四层：
1. **顶部页头**
   - 标题：单词熟练度统计
   - 副标题：展示当前用户单词掌握状态、复习打分与近 7 天复习活跃度
   - 右侧操作：`去今日复习`、`查看单词列表`
2. **核心指标区**
   - 保留旧 4 张 KPI 卡
   - 新增一个可前端推导的“熟练率 = masteredWords / activeWords”小指标，帮助判断整体状态
3. **学习质量区**
   - 平均连续记对次数
   - 平均简易度因子
   - 可补充说明文字，解释这两个数值的意义，避免用户只看数字不懂含义
4. **图表区**
   - 熟练度分层
   - 评分分布
   - 最近 7 天趋势
   - 图下增加简短说明文案，而不是只有图本身

### 5.3 视觉与样式方向
- 用 `Shadcn UI + Tailwind CSS` 重做，不沿用旧 Arco 卡片堆叠视觉。
- 风格建议偏“学习仪表盘”而不是“后台统计报表”：
  - 顶部 KPI 卡统一间距和强调色
  - 图表卡片采用同一边框/圆角/留白体系
  - 用颜色表达掌握度，但控制在 1~2 组主题色，不做杂色大屏风格
- 需要 Skeleton、Error、Empty 三种明确状态，不要只在控制台报错。

### 5.4 数据流与状态管理
nquiz 当前仓库只有 Next.js 骨架和 Query Provider：
- `src/app/layout.tsx`
- `src/components/providers/app-providers.tsx`
- `src/components/providers/query-provider.tsx`
- `src/lib/query/query-keys.ts`

因此这个页面在 nquiz 中属于 **净新增迁移实现**。建议：
1. 用 TanStack Query 拉取 dashboard 数据；
2. 新增 query key，例如 `vocabularyProficiency.dashboard`；
3. 由 server route / BFF 或 server action 对接后端统计接口，不在页面组件里直连散落请求；
4. 图表层优先使用声明式 React 图表封装，避免继续写大量 `ref + init + dispose` 命令式逻辑。

### 5.5 交互方式
建议补齐旧页面缺失的业务闭环：
1. 点击“今日待复习”卡片 -> 跳到单词复习入口；
2. 点击“活跃单词/已熟练”卡片 -> 跳到单词列表并带上相应筛选；
3. 图表 hover 展示解释性 tooltip；
4. 页面支持手动刷新；
5. 无数据时明确提示“先去创建单词或完成复习”。

## 6. 能力范围建议
### 首版必须覆盖
1. 当前用户维度的单词掌握概览；
2. 熟练度分层分布；
3. 复习评分分布；
4. 近 7 天复习趋势；
5. 与单词列表 / 今日复习页面的跳转联动。

### 可优化但不要求首版完成
1. 标签维度/词库维度筛选；
2. 时间范围切换（7 天 / 30 天）；
3. 导出 PNG/CSV；
4. 多用户/班级对比；
5. 复习评分维度重构（比如从 0/3/5 收敛到“忘记/模糊/熟练”三类统计）。

### 建议延后
1. 跨主题统一统计中心；
2. 复杂 BI 透视；
3. 团队榜单或教师视角分析。

## 7. 风险点
1. **接口复用边界**：旧接口是 `/api/statistics/themes/vocabulary-proficiency/dashboard`，如果 nquiz 后端不直接兼容该路径，需要额外 BFF/服务层适配。
2. **数据口径是用户私有的**：当前 SQL 全按 `create_user` 过滤。若未来产品想做班级/团队视角，需要重新设计数据权限和聚合逻辑。
3. **评分桶语义不一致**：统计按 0~5，当前前端复习交互只给 0/3/5，可能导致统计图形态和用户认知不完全一致。
4. **页面入口重定向风险**：旧系统后期已经删除 statistics-center 菜单，nquiz 若重新恢复旧路径，会把历史包袱重新带回来。
5. **nquiz 目前是骨架仓库**：当前还没有 vocabulary 业务模块，迁移这个页面时必须同步规划 vocabulary domain 的路由归属与 API 组织方式，不能只孤立造一张图表页。

1. 打开 nquiz 的单词熟练度统计页，可稳定看到 4 个核心指标、2 个学习质量指标、3 张图表；
2. 数据口径与旧后端 `VocabularyProficiencyStatisticsServiceImpl` 一致：
   - 已熟练 = 未归档且 repetition >= 6
   - 今日待复习 = 未归档且 nextReviewDate <= now
   - 分层区间 = 0 / 1-2 / 3-5 / 6+
3. 页面具备 loading / empty / error 三态；
4. 可以从该页进入“今日复习”或“单词列表”，形成闭环；
5. 不再依赖旧 `statistics-center` 作为入口前提。
