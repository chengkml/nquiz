1. 目标与结论
- 该需求对应 quiz 的 `MCP 服务器管理页`，原前端入口在 `frontend/src/pages/McpServer/index.tsx`，原路由 `/frame/mcp-server`，后端主入口在 `backend/src/main/java/com/ck/quiz/mcp/controller/McpServerController.java`，核心服务在 `backend/src/main/java/com/ck/quiz/mcp/service/impl/McpServerServiceImpl.java`。
- 这个页面不是单纯的“配置表维护页”，而是 MCP 能力接入链路的上游入口：负责维护 MCP Server 基础连接信息、执行健康检查、握手初始化、发现工具，并把发现结果沉淀到 `mcp_tool` 表，供后续工具管理与 Agent 绑定使用。
- nquiz 重构时应保留 3 个核心能力：`服务器台账管理`、`连通性/健康检查`、`工具发现入口`；不建议继续沿用旧版“点击名称打开工具抽屉 + 在抽屉中直接加载工具”的耦合交互，建议重构为 Next.js App Router 下的“列表 + 详情侧板/详情页 + 明确的发现动作”结构。

2. 页面定位与功能边界
- 页面定位：后台系统中的 MCP 基础设施管理页面，负责管理外部 MCP Server 的接入元数据与可用性状态。
- 功能边界（基于真实代码）
  - 已实现并应保留：
    - 查询分页：`searchMcpServers` -> `/mcp/server/search`
    - 新增服务器：`createMcpServer` -> `/mcp/server/create`
    - 编辑服务器：`updateMcpServer` -> `/mcp/server/update`
    - 删除服务器：`deleteMcpServer` -> `/mcp/server/delete/{id}`
    - 健康检查：`healthCheckMcpServer` -> `/mcp/server/{id}/health-check`
    - 发现工具：`listDiscoveredTools` -> `/mcp/server/{id}/discovered-tools`
  - 页面未真正闭环但后端语义已存在：
    - 工具发现后写入 `mcp_tool` 表。`McpServerServiceImpl.discoverToolsOnce` 内部会调用 `saveMcpTool(...)` 自动落库，并设置 `status=REGISTERED`。
    - 工具调用能力已存在：`POST /api/mcp/server/call-tool`，但当前 server 管理页没有调用入口。
  - 历史包袱/不一致点：
    - 前端声明了 `importMcpTools` API：`/mcp/server/{id}/tools/import`，但当前 `McpServerController` 中未看到对应接口，且页面 `handleLoadAllTools` 只是提示“成功加载 X 个工具”并清理缓存，没有真正调用导入接口；说明“加载工具”文案与真实行为不一致。
    - 搜索参数中前端传了 `env`，但 `McpServerQueryDto` 只有 `status`，后端 `search` 也未按 env 过滤，属于遗留噪音参数。

3. 真实代码定位与当前实现
- 前端入口
  - `../quiz/frontend/src/pages/McpServer/index.tsx`
  - `../quiz/frontend/src/pages/McpServer/api/index.ts`
  - `../quiz/frontend/src/router/index.tsx` 中 `path: "mcp-server"`
- 后端入口
  - `../quiz/backend/src/main/java/com/ck/quiz/mcp/controller/McpServerController.java`
  - `../quiz/backend/src/main/java/com/ck/quiz/mcp/service/McpServerService.java`
  - `../quiz/backend/src/main/java/com/ck/quiz/mcp/service/impl/McpServerServiceImpl.java`
  - `../quiz/backend/src/main/java/com/ck/quiz/mcp/entity/McpServer.java`
  - `../quiz/backend/src/main/java/com/ck/quiz/mcp/entity/McpTool.java`
- 当前数据结构
  - `mcp_server`：`name`、`identifier`、`description`、`address`、`authConfig`、`status`、`lastHeartbeatAt`
  - `mcp_tool`：`serverId`、`env`、`originName`、`displayName`、`description`、`category`、`tags`、`status`、`schemaJson`、`strategyJson`、`visibilityJson`、`sourceDeletedFlag`
- 当前交互链路
  - 列表页首次加载即查分页数据。
  - 点击“健康检查”后，后端执行 `handshake -> initialize`，成功则更新 server `status=ACTIVE`、`lastHeartbeatAt=now`，失败则置为 `INACTIVE`。
  - 点击服务器名称后打开工具抽屉，调用 `/discovered-tools`；后端再次执行 `handshake -> initialize -> tools/list`，并在发现过程中将 tool 自动写入 `mcp_tool` 表。
- 当前服务实现特征
  - 后端使用 SSE/JSON-RPC 风格链路与 MCP Server 通信，包含重试、超时、初始化通知、tools/list、tools/call。
  - `authConfig` 被直接当作 Bearer Token 头使用，当前是纯文本保存与回传，敏感信息治理较弱。
  - `update` 做了“仅创建人可更新”校验，但 `search` 是全表查询，删除/查看权限边界在该页代码中不明确。

4. nquiz 重构建议（Next.js 语境）
- 路由与信息架构
  - 建议使用 App Router，路由形态改为：
    - `/mcp/servers`：服务器列表页
    - `/mcp/servers/[id]`：服务器详情/工具发现结果页（或右侧详情抽屉）
  - 不建议把“查看工具”绑定在名称点击上，建议显式拆成操作：`详情`、`健康检查`、`发现工具`。
- 页面信息架构
  - 顶部：页面标题 + 模块说明 + 新增服务器按钮。
  - 列表筛选：关键字、状态。
  - 列表字段：名称、标识、地址、状态、最近心跳、创建时间、操作。
  - 详情区：基础配置、认证状态（仅显示“已配置/未配置”，默认不回显明文）、最近发现时间、最近发现结果摘要。
  - 工具发现区：最近一次发现到的工具列表、schema 摘要、与已入库工具的差异统计（新增/更新/失效）。
- 视觉设计方向
  - 使用 Shadcn UI 的 `Card`、`Data Table`、`Sheet/Drawer`、`Dialog`、`Badge`、`Alert`、`Tabs`、`Tooltip`。
  - 状态采用语义色 Badge：`CREATED / ACTIVE / DEGRADED / INACTIVE`。
  - 健康检查与发现工具使用带 loading 的异步操作按钮，结果以内联反馈 + toast 呈现，不再依赖大面积弹窗。
- 表单与样式体系
  - 表单使用 `React Hook Form + Zod`，对 `name/identifier/address/authConfig` 做统一校验。
  - 样式使用 Tailwind CSS，就近写在组件中，不再保留旧版 less 页面样式方案。
- 数据流与状态管理
  - 列表、详情、发现结果使用 TanStack Query 管理。
  - `健康检查`、`新增/编辑/删除`、`发现工具` 走 Mutation，成功后精准失效 `servers`、`server-detail`、`server-tools` 查询键。
  - 对“发现工具”建议增加服务端任务态或至少保留最近结果快照，避免每次打开页面都重新握手。

5. 能力保留 / 优化 / 重组 / 延后
- 必须保留
  - 服务器新增、编辑、删除、分页查询
  - 健康检查与状态回写
  - 工具发现入口，以及发现结果沉淀到工具域的能力
- 建议优化
  - 明确把“发现工具”作为独立动作，不再藏在“点名称”里
  - 认证配置默认脱敏显示，不回显完整 token
  - 健康状态与发现结果增加最近执行时间、错误摘要
  - 补齐权限模型：至少区分查看、编辑、删除、执行检查
- 建议重组
  - `MCP 服务器管理` 专注 server 连接与发现；`MCP 工具管理` 专注工具元数据、启停、分类、可见范围、Agent 绑定
  - 页面间通过 serverId / toolCount / lastDiscoveryAt 建立导航关系，而不是在一个抽屉里做半闭环
- 可延后
  - 在线 `call-tool` 调试台
  - 差异比对（本次发现 vs 上次发现）
  - 批量健康检查、批量发现
  - 更复杂的多认证方式（非 Bearer）

6. 风险点
- 安全风险：`authConfig` 当前明文存储且 DTO 会直接回传，nquiz 需要至少做脱敏展示，理想情况应改成密文存储或密钥引用。
- 协议风险：当前实现强依赖 SSE/JSON-RPC 交互时序，服务端超时、半开连接、非标准返回都可能导致状态误判。
- 语义风险：server 页中的“加载工具”与真实后端行为不一致，迁移时如果照搬会继续扩大认知偏差。
- 权限风险：当前 `search` 是全量可见，但 `update` 仅创建者可改，查看/删除/执行动作口径并不统一，nquiz 需要统一权限设计。
- 数据模型风险：`mcp_tool.env` 实际被写成 `server.identifier`，字段命名与业务含义不一致，后续若继续扩展环境维度会混淆。

7. 未决问题
- nquiz 是否仍沿用“每次发现即自动写入工具表”的策略，还是改成“发现结果预览后确认同步”。
- `identifier` 在 nquiz 中到底代表服务器唯一编码、运行环境，还是租户/来源域；当前旧实现存在语义混用。
- 是否需要在 nquiz 中提供“最近一次健康检查日志 / 发现日志”，便于定位 SSE 握手失败原因。
- 工具域是否继续复用旧版 `REGISTERED/ENABLED/DISABLED/GRAY_RELEASE/SOURCE_REMOVED` 状态体系。

8. 建议验收标准
- 能在 `/mcp/servers` 完成分页查询、关键字筛选、新增、编辑、删除。
- 健康检查成功时可刷新 server 状态与最近心跳；失败时展示明确错误反馈。
- 能对某个 server 触发工具发现，并看到返回的工具列表摘要。
- 发现结果能够稳定沉淀到工具域，且可从工具管理或 Agent 选工具链路中被消费。
- 页面不回显完整认证敏感信息，至少做到脱敏展示。
- UI 与状态管理符合 nquiz 技术栈：Next.js App Router + Tailwind + Shadcn UI + RHF/Zod + TanStack Query。

9. 对 nquiz 当前现状的判断
- 当前 nquiz 仓库仍是骨架阶段，实际只有 `src/app/page.tsx` 首页、Provider 与少量基础代码，尚无 MCP 模块实现。
- 因此本需求应按“新功能菜单迁移分析”处理，而不是做旧页面逐行翻译；推荐先完成 server 管理与 tool 管理的模块拆分设计，再分阶段落地。
