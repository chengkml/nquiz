## 1. 目标
- 本需求对应 quiz 的“知识点统计页”，本质上不是独立 BI 模块，而是 **Knowledge 学习模块的掌握度观测面板**。
- 迁移到 nquiz（Next.js 重构版）时，目标不是机械复刻旧的 `statistics-center/knowledge-mastery`，而是在保留知识点学习语义的前提下，重构为更贴近 Knowledge 域的学习洞察页。
- 首版应保留旧页面的核心统计能力：知识点规模、活跃/待复习/已掌握概览、掌握分层、学科分布、复习评分分布、近 7 天复习趋势，并补上与“知识点列表 / 今日复习”的业务闭环。

### 2.4 旧页面真实展示内容
旧页面固定展示以下内容：
1. 4 个 KPI：
   - 知识点总数 `totalKnowledges`
   - 活跃知识点 `activeKnowledges`
   - 今日待复习 `dueTodayKnowledges`
   - 已掌握 `masteredKnowledges`
2. 2 个均值：
   - 平均连续记对次数 `averageRepetition`
   - 平均简易度因子 `averageEasinessFactor`
3. 4 个图表：
   - 知识掌握分层分布 `masteryDistribution`
   - 学科知识点分布 `knowledgeCountBySubject`
   - 复习评分分布 `reviewScoreDistribution`
   - 近 7 天复习趋势 `reviewCountByLastSevenDays`

### 2.5 旧后端真实统计口径
`KnowledgeMasteryStatisticsServiceImpl` 已把统计规则写死：
1. **仅统计当前登录用户**：知识点按 `knowledge.create_user = 当前用户` 过滤。
2. **总数 / 活跃 / 归档**：来源于 `knowledge` 表和 `ReviewModel.archived`。
3. **已掌握**：未归档且 `repetition >= 6`。
4. **今日待复习**：未归档且 `next_review_date <= now`。
5. **平均连续记对次数 / 平均简易度因子**：只统计未归档知识点。
6. **掌握分层**：
   - 0 次连对：未掌握
   - 1-2 次连对：初步掌握
   - 3-5 次连对：稳定掌握
   - 6+ 次连对：深度掌握
7. **学科分布**：按 `subject.name` 聚合，空值回落到“未归类学科”。
8. **复习评分分布**：从 `review_log` 统计 0~5 分各桶。
9. **近 7 天趋势**：统计近 7 天每天复习次数。

### 2.6 与知识点模块的真实业务关系
这张统计页并不是孤立存在，而是 Knowledge 模块学习闭环的聚合视图：
- `Knowledge/index.tsx` 提供知识点 CRUD、归档、重置、学科分类筛选、知识点详情、生成题目等入口；
- `Knowledge/Review.tsx` 通过 `getDueToday()` 和 `reviewKnowledge()` 执行今日复习；
- `Knowledge` 实体继承 `ReviewModel`，天然具备 `repetition / nextReviewDate / easinessFactor / archived / totalReviewCount / lastScore` 等学习状态字段；
- 所以“知识点统计页”本质上是 **知识点学习状态的二次聚合展示**。

## 3. 范围
### 3.1 首版应纳入范围
1. 当前用户维度的知识点掌握概览；
2. 掌握分层分布；
3. 学科维度分布；
4. 复习评分分布；
5. 最近 7 天复习趋势；
6. 与知识点列表页、今日复习页的跳转联动；
7. 明确的 loading / empty / error 三态。

### 3.2 建议保留的旧能力
1. 以 `ReviewModel` 为基础的掌握度口径；
2. 用户私有视角而非全局视角；
3. 学科分布统计；
4. 复习评分与复习趋势统计。

## 4. 非范围
1. 首版不要求恢复旧的 `statistics-center` 总入口；
2. 不要求做跨用户 / 班级 / 团队统计；
3. 不要求首版支持复杂筛选（按分类、时间范围、标签、题目来源等）；
4. 不要求首版提供导出、打印、报表订阅；
5. 不要求把旧 Arco + ECharts 命令式实现原样搬进 nquiz；
6. 不要求在本需求中顺手重构整个 Knowledge 模块全部页面。

## 5. 实现思路（nquiz / Next.js 语境）
### 5.1 页面定位
建议不要恢复 `statistics-center/knowledge-mastery` 的旧入口，而是把它迁到 Knowledge 域下，例如：
- `app/(dashboard)/knowledge/mastery/page.tsx`
或按最终 IA 归入 `learning/knowledge/mastery`。

理由：
- 旧系统已经删除统计中心菜单，说明“独立统计中心”不是必须继承的产品约束；
- 从代码上看，它本来就是 Knowledge 模块的附属观察面板；
- 在 nquiz 里按功能域挂载，比继续保留历史壳路由更清晰。

### 5.2 信息架构建议
建议页面拆成四层：
1. **页头区**
   - 标题：知识点掌握统计
   - 副标题：展示当前用户知识点掌握状态、学科分布与复习趋势
   - 右侧动作：`去今日复习`、`查看知识点列表`
2. **核心指标区**
   - 知识点总数
   - 活跃知识点
   - 今日待复习
   - 已掌握
   - 可新增前端派生指标：掌握率 = 已掌握 / 活跃知识点
3. **学习质量区**
   - 平均连续记对次数
   - 平均简易度因子
   - 配解释性文案，避免用户不知道两个数代表什么
4. **图表区**
   - 掌握分层
   - 学科分布
   - 评分分布
   - 近 7 天趋势

### 5.3 技术实现建议
基于当前 nquiz 仓库现状（仅有 Next.js 骨架、Query Provider、基础目录，暂无 Knowledge 业务模块），建议按新实现落地：
1. 页面层使用 App Router；
2. 数据请求走 TanStack Query，而不是页面内手搓 `useEffect + useState`；
3. 新增 query key，例如 `knowledgeMastery.dashboard`；
4. 通过 server route / BFF / server action 适配旧后端统计接口，不把请求逻辑散落在客户端组件里；
5. UI 使用 Shadcn UI + Tailwind CSS；
6. 图表优先采用更声明式的 React 封装，不延续旧页面里多个 `echarts.init / dispose` 的命令式写法；
7. 状态上补齐 Skeleton、Empty、Error 与手动刷新能力。

### 5.4 交互补强建议
旧页主要问题是“只看不动”。nquiz 应补齐闭环：
1. 点击“今日待复习” -> 跳转到知识点复习页；
2. 点击“活跃知识点 / 已掌握” -> 跳到知识点列表并带预设筛选；
3. 图表 hover 提供口径解释；
4. 无知识点时提示先创建知识点；
5. 无待复习时提示今日任务已完成。

## 6. 影响点
1. **路由层**：需要在 nquiz 新增 Knowledge 统计页路由，不建议恢复旧 `statistics-center`；
2. **数据层**：需要定义 dashboard DTO / query key / API 适配层；
3. **Knowledge 主链路**：统计页要与知识点列表页、复习页建立跳转关系；
4. **权限与菜单层**：若 nquiz 后续有菜单系统，建议把入口挂到 Knowledge 域，而不是独立统计中心；
5. **图表与组件层**：需要统一到 nquiz 的 UI 体系，而不是继续使用旧 Arco 页面样式。

## 7. 风险 / 疑问
1. **后端接口兼容问题**：旧接口路径是 `/api/statistics/themes/knowledge-mastery/dashboard`，nquiz 是否直接复用旧后端，还是在 Next.js 侧做 BFF 适配，需要尽快确认。
2. **评分桶语义问题**：统计按 0~5 分完整分布，但知识点复习页当前前端只提供 0 / 3 / 5 三档快捷评分，这会导致 1 / 2 / 4 桶可能长期为空或仅来自历史数据。nquiz 需要决定是继续保留完整 0~5，还是收敛为更贴合交互的三档展示。
3. **入口归属问题**：是挂在 `knowledge`，还是放进更泛化的 `learning insights`，需要产品侧最终拍板；但不建议恢复旧统计中心路径。
4. **仓库现状问题**：nquiz 当前还是骨架仓库，Knowledge 模块本身尚未迁入，因此这张统计页不能孤立建设，至少要同步规划它和 Knowledge 列表/复习模块的关系。
5. **用户维度限制**：旧实现只看当前用户；若后续需要教师/团队视角，统计口径、权限模型与聚合方式都要重做。

## 8. 建议验收标准
1. 打开 nquiz 的知识点掌握统计页后，可以稳定看到：
   - 4 个核心指标；
   - 2 个学习质量指标；
   - 4 个图表（掌握分层、学科分布、评分分布、近 7 天趋势）。
2. 数据口径与旧后端 `KnowledgeMasteryStatisticsServiceImpl` 保持一致：
   - 已掌握 = 未归档且 `repetition >= 6`
   - 今日待复习 = 未归档且 `nextReviewDate <= now`
   - 掌握分层 = 0 / 1-2 / 3-5 / 6+
3. 页面具备 loading / empty / error 三态；
4. 可以从该页进入知识点列表页与知识点复习页，形成业务闭环；
5. 页面归属在 Knowledge 域下，而不是强依赖旧的 `statistics-center`；
6. UI 与交互符合 nquiz 技术栈风格，不再复刻旧的 Arco 报表页样式。

## 9. 结论
- **应保留**：知识点掌握概览、掌握分层、学科分布、评分分布、近 7 天趋势这些核心学习洞察能力；
- **应优化**：入口归属、信息架构、状态管理、图表实现、与知识点列表/复习页的跳转闭环；
- **可延后**：复杂筛选、导出、跨用户对比、跨主题统一统计中心。

整体判断：该需求适合在 nquiz 中按 **Knowledge 域学习洞察页** 落地，而不是恢复旧统计中心壳。
