结论：该需求本质上不是单纯“做一个模型配置表”，而是把 quiz 里的 LLM 模型管理能力迁移到 nquiz，并在 Next.js 语境下重构为“模型注册 + 默认模型切换 + 按类型调试验证”的完整管理工作台。建议在 nquiz 中按“AI 基础设施/模型管理”模块落地，而不是机械复刻旧的 `/frame/llmmodel` 页面。首版必须保留模型 CRUD、按类型默认模型、文本模型聊天测试、视觉/图像模型多模态测试这四条主链路；历史命名不一致和表单字段错配问题应在迁移时一并清理。

一、目标
1. 在 nquiz 中承接 quiz 的 `LlmModel` 模块，建立可维护的模型管理能力，用于后续聊天、OCR、知识库、Agent 等能力复用。
2. 保留旧系统“不同模型类型独立配置、独立设默认、可直接测试”的业务语义。
3. 借迁移机会修正旧实现中的命名不一致、字段映射错误、页面职责分散等历史包袱。
4. 方案默认站在 nquiz 的技术基线下设计：Next.js App Router + Tailwind CSS + shadcn/ui + React Hook Form + Zod + TanStack Query，后端数据层优先按新仓库规范重建，而不是照搬旧 Spring + Arco 的页面实现。

三、旧系统真实功能与业务语义
1. 页面核心能力不是单一列表，而是四块：
   - 模型列表分页查询
   - 模型新增/编辑/删除
   - 同类型默认模型设置
   - 按模型类型进行能力测试（TEXT -> 聊天；VISION/IMAGE -> 图文测试）
2. 旧页面支持的模型类型为：`TEXT`、`VISION`、`VOICE`、`IMAGE`、`EMBEDDING`。
3. 数据字段来自 `LLMModel` 实体，核心字段包括：
   - `name`
   - `provider`
   - `type`
   - `descr`
   - `apiKey`
   - `apiEndpoint`
   - `contextWindow`
   - `inputPricePer1k`
   - `outputPricePer1k`
   - `isDefault`
   - `config`
4. 默认模型语义不是全局唯一，而是“按类型唯一”。`LLMModelServiceImpl#setDefaultModel` 会先把同类型模型清空默认，再把目标模型置为默认。
5. 文本测试并不是页面内手搓推理，而是复用通用聊天接口 `/api/chat/stream`，通过 `config.modelName = model.name` 指定具体模型。
6. 多模态测试走专用接口 `/api/llm-model/{id}/test-multimodal`，上传 `prompt + image`，后端再用指定模型调用 Spring AI 的 `OpenAiChatModel` 做图文理解。
7. `LLMModelServiceImpl#getChatModel` / `getEmbeddingModel` 说明该模块不是孤立页面，而是系统内统一模型注册表；其他模块会依赖这里解析默认模型或按名称解析模型。
8. 后端搜索 SQL 还联表了 tag 关系，说明模型在旧系统里具备标签扩展能力，但当前前端页面并未真正把标签作为主交互展示。

四、已识别的历史包袱/不一致点
1. **描述字段命名不一致**：后端字段是 `descr`，但旧前端表单和提交 payload 用的是 `description`。从 `index.tsx` 可见新增和编辑都没有把描述映射到 `descr`，这会导致描述无法正确回写或回显。
2. **名称搜索字段错配**：前端搜索表单提交的是 `name`，但后端 `LLMModelServiceImpl#search` 实际读取的是 `queryDto.getKeyWord()`，不是 `name`。这意味着旧页面“按名称搜索”大概率并不真正生效，属于可借迁移修正的缺陷。
3. **页面职责偏散**：列表、配置、文本测试、多模态测试塞在同一页，通过多个弹窗/抽屉切换，功能能用，但信息层级一般，后续扩展供应商参数时会更乱。
4. **安全语义偏弱**：旧实体直接保存 `apiKey` 明文，前端编辑时也直接暴露可修改输入框。nquiz 迁移时至少应做“掩码展示 + 留空不改 + 服务端安全存储策略预留”。
5. **命名和文案粗糙**：旧页面存在“模形”等错字、`descr/description` 双轨、`LlmModel/LLMModel/llmmodel` 混用，迁移时应统一为一套领域命名。

五、范围
建议把本需求的一期范围定义为：
1. 在 nquiz 落地“模型管理”主页面，提供列表、筛选、分页、创建、更新、删除。
2. 支持模型类型枚举与按类型唯一默认模型设置。
3. 支持核心字段维护：名称、供应商、类型、API Key、API Endpoint、上下文窗口、输入/输出单价、描述、扩展配置。
4. 支持 TEXT 模型的基础聊天调试能力。
5. 支持 VISION / IMAGE 模型的单图多模态调试能力。
6. 建立可被其他模块复用的模型数据层与调用解析层（至少支持按名称取模型、按类型取默认模型）。
7. 完成命名修正：统一 `descr -> description` 或统一内部域模型字段，但前后端对外契约必须一致。

六、非范围
以下内容不建议作为本需求一期必做项：
1. 完整聊天中心迁移（会话列表、历史消息管理、删除会话等）。
2. 供应商特定的高级参数面板（temperature、top_p、max_tokens、seed、response_format 等精细配置）。
3. 批量导入/导出模型。
4. 标签体系 UI 化。
5. 语音模型、图片生成模型的专门工作台。
6. 模型调用审计、配额统计、计费报表。
7. 多租户/团队级模型权限隔离（如当前 nquiz 尚未建立完整权限体系，可后置）。

七、实现思路（nquiz 语境）
1. **页面定位**
   - 不建议继续沿用旧的 `/frame/llmmodel` 信息架构。
   - 建议在 nquiz 新导航下落为 `AI / Models` 或 `基础设施 / 模型管理` 页面，例如：`src/app/(dashboard)/ai/models/page.tsx`。
   - 若后续要兼容旧路径，再做 redirect，而不是继续沿用旧框架式命名。
2. **信息架构**
   - 顶部：页面标题、说明、创建按钮。
   - 左/上区域：筛选条（名称、提供商、类型、默认状态）。
   - 主区域：数据表格，展示名称、提供商、类型、默认状态、端点、上下文窗口、价格、更新时间。
   - 侧边 Sheet / Dialog：新增与编辑表单。
   - 详情调试区：建议合并成统一“模型调试面板”，根据类型切换文本调试或多模态调试，而不是保留两个完全独立抽屉。
3. **组件拆分建议**
   - `models-page.tsx`：页面编排
   - `model-table.tsx`：列表渲染
   - `model-form-dialog.tsx`：创建/编辑表单
   - `model-debug-sheet.tsx`：模型调试
   - `use-models-query.ts` / `use-model-mutations.ts`：查询与变更 hooks
4. **表单与校验**
   - 使用 `React Hook Form + Zod`。
   - 对 `apiEndpoint`、价格、上下文窗口、类型做明确 schema 校验。
   - 编辑时 `apiKey` 采用“留空表示不修改”的语义，避免旧页面那种明文回显与误覆盖。
5. **数据层建议**
   - nquiz 若采用 Drizzle，新表物理名应遵守仓库约定，建议使用 `quiz_llm_model`，不要直接继续使用旧表名 `llm_model`。
   - 构建统一领域模型：`ModelType` 枚举、`ModelRecord` 类型、`resolveDefaultModelByType(type)`、`resolveModelByName(name)` 等服务方法。
   - 默认模型切换需通过事务或唯一约束保证“同类型仅一个默认项”。
6. **请求与状态管理**
   - 列表/详情走 TanStack Query。
   - 创建/更新/删除/设默认走 mutation，并在成功后失效对应 query key。
   - 可新增类似 `queryKeys.models.list(filters)`、`queryKeys.models.detail(id)` 的 key，而不是继续把模型状态散落在页面里。
7. **模型调试实现**
   - TEXT：复用统一 chat service / route handler，支持流式返回；不建议把完整聊天产品能力塞进模型管理页，只保留“单轮/短上下文调试”。
   - VISION/IMAGE：复用 multipart 上传接口，支持单图 + prompt 测试，返回最终结果即可；若要流式可二期增强。
8. **安全与展示策略**
   - 页面列表不直接展示完整 API Key。
   - 编辑弹窗显示“已配置 / 未配置”与掩码占位。
   - 后端保留安全存储升级空间（至少在服务层隔离，不把明文到处透传）。

八、影响点
1. **路由与导航**：需要新增 nquiz 模型管理入口；如果已有 Dashboard/Sidebar，也要补菜单归类。
2. **数据库 schema**：需要新增模型表及可能的唯一索引、枚举字段；若后续兼容标签，还会关联标签关系表。
3. **服务层**：后续 Chat、OCR、Knowledge、Agent 等模块可能都会依赖这里解析默认模型，因此模型服务要做成基础设施层，不应只为页面临时拼接口。
4. **鉴权与权限**：旧系统依赖登录态，nquiz 当前骨架里还看不到正式鉴权实现。若没有权限体系，页面虽能开发，但实际上线前必须明确谁可以查看/修改密钥。
5. **流式能力**：文本调试如果保留流式输出，会影响 route handler / 代理层 / 前端流读取逻辑。
6. **文件上传链路**：多模态测试需要处理图片大小、MIME 类型、上传错误与服务端超时。
7. **字段兼容**：旧系统已有 `descr`、`config`、`isDefault` 等字段语义；迁移时需统一 DTO / schema / form 命名，避免旧问题继续带入。

九、风险 / 疑问
1. **nquiz 当前基础设施不足**：当前仓库还没有正式业务 API、鉴权、模型表 schema，本需求如果直接进入开发，实际会连带补一批基础设施，需控制边界。
2. **API Key 存储策略未定**：若直接照旧明文落库，安全风险较高；若要做加密/脱敏，又会影响编辑与测试流程，需要先定策略。
3. **默认模型语义要不要延续“按类型唯一”**：从旧实现看答案是“要”，但仍建议产品层确认，尤其是 IMAGE/VISION 是否需要拆开默认项。
4. **模型类型边界是否继续保留五类**：`VOICE`、`IMAGE` 在旧页面没有完整专门测试链路，只是作为配置类型存在；nquiz 首版可先保留枚举，但不必强行给每类都做专用工作台。
5. **多模态接口兼容性**：旧实现通过 Spring AI + OpenAI 兼容接口调用，不同供应商对 image input 格式支持并不完全一致，nquiz 若后端方案变化，需要重新验证兼容层。
6. **搜索/筛选口径需要统一**：旧实现已经暴露出 `name/keyWord` 错配，迁移时必须先定义清楚查询字段，不然前后端会继续对不上。

十、建议验收标准
1. 用户可以查看模型列表，并按名称、提供商、类型、默认状态完成真实可用的筛选与分页。
2. 用户可以新增模型，必填项校验正确；非法端点、空名称、非法价格、非法类型都有明确报错。
3. 用户可以编辑模型，并支持“留空不更新 API Key”；描述字段能正确回显与保存，不再出现 `descr/description` 错配。
4. 用户可以删除非受保护模型，并在交互上得到明确二次确认。
5. 用户可以把某个模型设置为所属类型的默认模型，且设置后同类型只保留一个默认项。
6. 对 TEXT 模型，用户可以在页面内发起基础聊天调试并看到流式或准实时响应。
7. 对 VISION / IMAGE 模型，用户可以上传单张图片与提示词完成多模态测试，并看到结果或错误提示。
8. API Key 在列表和编辑态默认不明文暴露；权限不足用户无法随意修改敏感配置。
9. 代码结构符合 nquiz 技术基线：App Router、Tailwind、shadcn、RHF + Zod、TanStack Query、可复用的数据层/服务层。

十一、能力取舍建议
1. **必须保留**
   - 模型 CRUD
   - 按类型默认模型
   - TEXT 调试
   - VISION/IMAGE 单图调试
   - 统一字段与校验
2. **建议优化**
   - 统一命名（description、defaultByType、modelType）
   - 调试入口统一为一个面板
   - API Key 掩码与安全策略
   - 真实可用的搜索筛选
3. **可重组或延后**
   - 标签 UI
   - 批量导入/导出
   - 高级供应商参数
   - 完整会话历史
   - 语音/图片生成专门工作台

十二、摘要
该需求应按“AI 模型基础设施迁移”处理，而不是孤立页面翻译。旧 quiz 的 `LlmModel` 实际承担了系统模型注册表职责，并为聊天与多模态能力提供模型配置、默认模型解析和即时调试。nquiz 首版建议围绕“模型管理 + 默认项 + 调试验证”做一个更清晰的 Next.js 工作台，同时修复旧系统中 `descr/description` 与 `name/keyWord` 这两类明确缺陷，为后续 Chat/OCR/Knowledge/Agent 模块复用打基础。
