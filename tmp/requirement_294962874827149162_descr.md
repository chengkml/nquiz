结论：`错题本页` 在 quiz 里不是通用题库，而是**当前登录用户自己的错题沉淀台账**，本质是“个人错题记录 + 分类检索 + 图片留档 + OCR 辅助录入”。迁移到 nquiz 时，应保留“个人隔离的错题列表、筛选、增删改、原图关联、OCR 回填”这条主链路；不建议照搬旧 Arco DataManager 页面，而应按 Next.js App Router + Tailwind + shadcn/ui 重构为更明确的“列表工作台 + 表单抽屉/对话框”结构。可保留 OCR 辅助录入，但要修正旧实现里上传即落库文件、权限语义偏混乱、OCR 覆盖策略偏粗糙的问题。

一、目标
- 在 nquiz 中迁移“错题本”菜单能力，形成一个面向当前用户的错题管理页。
- 保留核心业务语义：按学科/分类整理错题，记录题型、题干、答案、难度、备注，支持关联原始图片与 OCR 文本。
- 在实现方式上升级为更适合 Next.js 重构版的页面结构、状态管理和交互反馈，而不是复刻旧页面实现。

三、代码定位与当前实现
1. **页面定位与权限语义**
   - 旧系统把它作为菜单 `wrong-question` 暴露，并在 SQL 中给 `sys_mgr` 角色分配菜单权限。
   - 但后端查询 `WrongQuestionServiceImpl.search(...)` 又强制追加 `wq.create_user = :createUser`，说明页面展示的是“当前用户自己的错题”，不是全局共享错题库。
   - 这意味着它的真实语义更像“个人学习工作台”，而不是传统后台主数据页面；旧系统在菜单归属和数据隔离语义上有一点混搭。

2. **数据结构**
   - 实体字段来自 `WrongQuestion.java`：`subjectId`、`categoryId`、`type`、`content`、`answer`、`difficulty`、`remark`、`originalImageFileId`、`originalImageName`、`ocrText`。
   - DTO 暴露 `subjectName`、`categoryName`、`originalImageUrl`，用于列表展示与图片预览。
   - 表结构是轻量错题记录表，没有做题来源、掌握状态、复习次数、标签、关联题库题目等更复杂能力。

3. **列表与检索**
   - 前端支持按 `subjectId`、`categoryId`、`type`、`difficulty`、`content` 过滤，并分页查询。
   - 列表主要展示题目内容预览、学科、分类、题型、难度、原图缩略图、创建时间，以及编辑/删除操作。
   - `answer`、`remark`、`ocrText` 不在列表主表展示，只在编辑态可见。

4. **新增/编辑链路**
   - 新增与编辑共用一个 Modal 表单，字段包含学科、分类、题型、难度、题目内容、答案、解析/备注、原图/OCR。
   - 学科变化会触发分类列表重新加载，分类依赖 `/api/category/subject/{subjectId}`。
   - 保存时直接调用 `/wrong-question/create` 或 `/wrong-question/update`。

5. **OCR 与图片关联链路**
   - 前端先调用 `/api/file/upload` 上传原图，path 默认 `wrong-question/`。
   - 上传成功后立即调用 `/api/ocr/recognize`，并以 `text/event-stream` 方式按 SSE 流式读取识别结果。
   - 识别结果累计写入 `ocrText`；若题目内容 `content` 当前为空，则自动把 OCR 结果回填到内容框。
   - 最终保存错题记录时，把 `originalImageFileId`、`originalImageName`、`ocrText` 一并写入错题表。

6. **对外依赖**
   - 学科：`/api/subject/list`
   - 分类：`/api/category/subject/{subjectId}`
   - OCR 模型：`/llm-model/list-by-type/VISION`
   - 文件上传/下载：`/api/file/upload`、`/api/file/download`
   - OCR 服务本身依赖 `ocrRecognize` 提示词模板和视觉模型配置。

四、范围
本次 nquiz 迁移建议纳入 MVP 的范围：
1. 当前用户自己的错题列表页。
2. 按学科、分类、题型、难度、题干关键词筛选与分页。
3. 新增/编辑/删除错题。
4. 错题基础字段维护：学科、分类、题型、题干、答案、难度、备注。
5. 原始图片上传、预览、回显。
6. OCR 识别结果展示，并支持把识别内容带入题干编辑。
7. 保留个人数据隔离语义，不能误做成全员共享后台表。

五、非范围
本次分析不建议把以下内容作为首版必做：
1. 错题复习计划、掌握度、复训队列、订正记录。
2. 与练习、考试、题库题目的一体化关联回溯。
3. 批量导入/批量 OCR/批量导出。
4. 多图上传、图片裁剪、图片标注。
5. 结构化 OCR 解析（题干/答案/解析分段提取）。
6. 错题分享、团队协作、全局管理视角。

六、实现思路（nquiz / Next.js 语境）
1. **路由与页面组织**
   - 建议新路由使用更明确的复数资源语义，例如：`app/(dashboard)/study/wrong-questions/page.tsx`。
   - 如需兼容旧链接，可对 `/frame/wrong-question` 做 redirect，而不是继续沿用旧式 frame 路由结构。

2. **信息架构**
   - 页面采用“顶部筛选区 + 主列表区 + 右侧抽屉/弹窗编辑”的工作台布局。
   - 列表保留题干预览、学科、分类、题型、难度、原图、更新时间/创建时间；详情编辑放入 Drawer/Dialog，避免主表过载。
   - 原图与 OCR 结果放在表单下半区，形成“上传 → 识别 → 修订题干”的录入闭环。

3. **前端状态管理**
   - 筛选与表单：`React Hook Form + Zod`
   - 列表查询与增删改：`TanStack Query`
   - 分类选项：依赖 `subjectId` 的条件查询
   - OCR 流式识别：独立成 `useOcrStream` hook 或 server action / route handler 代理，不把 SSE 解析逻辑直接堆在页面组件里。
   - Query Key 建议区分：`wrongQuestions.list`、`wrongQuestions.detail`、`wrongQuestions.meta.subjects`、`wrongQuestions.meta.categories`。

4. **交互优化建议**
   - OCR 识别结果不要默认粗暴覆盖用户已输入内容；应在“内容为空时自动回填，否则提示追加/覆盖/仅保留 OCR 结果预览”。
   - 图片上传成功但未保存时，要给出“未保存图片不会进入错题记录”的明确提示，或改为保存时再最终落档。
   - 编辑态中如果已有原图，应支持替换图片，而不是只能清空后重传。

5. **后端/数据层建议**
   - 若 nquiz 走独立新表，按当前工程约定建议使用 `quiz_wrong_question` 之类带 `quiz_` 前缀的新表名，而不是直接新增无前缀物理表。
   - API 语义上保留“只查当前用户自己的错题”，不要在 Next.js 重构时无意改成管理员可见所有人数据。
   - 如果文件与 OCR 依然复用现有 quiz 后端，需要补充一个更稳定的 BFF/代理层，处理鉴权、SSE 与错误标准化。

七、影响点
1. **路由与菜单**：需要在 nquiz 新导航体系中给错题本定义新的菜单归属；建议按“学习 / 题库工具”归类，而非单纯后台管理。
2. **数据模型**：错题表需要继续承载图片元数据和 OCR 文本；若采用新表命名，要评估与旧库迁移脚本兼容性。
3. **依赖接口**：学科、分类、OCR 模型、文件服务、OCR 服务都要接入，属于多接口组合页面。
4. **鉴权与隔离**：必须保留基于当前登录用户的数据隔离；这是该页面最关键的业务语义之一。
5. **文件存储**：原图上传与下载链路直接影响用户录入体验，也影响历史数据可追溯性。
6. **SSE/流式能力**：Next.js 侧若做代理，需要确保不被缓冲，否则 OCR 体验会退化成“长时间无响应后一次性返回”。

八、风险 / 疑问
1. **用户语义 vs 菜单权限语义冲突**：旧实现菜单给 `sys_mgr`，但数据是“仅本人可见”；nquiz 中要明确它是所有登录用户可用的个人学习页，还是管理后台中的个人工具页。
2. **表命名策略**：nquiz 新增物理表要求 `quiz_` 前缀，而旧模块使用的是 `wrong_question`；若做独立落库，需要明确是否迁移旧数据、是否保留兼容层。
3. **文件孤儿风险**：当前是“先上传文件，再决定是否保存错题”，如果用户取消，可能留下未关联文件。
4. **OCR 强依赖外部配置**：视觉模型列表、`ocrRecognize` 提示词模板缺失时，页面虽然能打开，但 OCR 功能会失效。
5. **OCR 回填精度有限**：当前只是把识别文本整体塞回题干，不区分题干/答案/解析，复杂图片题可能需要人工大量修订。
6. **分类依赖学科**：如果后续想支持跨学科错题或更灵活标签体系，现有 `subject -> category` 单路径模型可能偏紧。

九、建议验收标准
1. 登录用户可进入 nquiz 错题本页，并只看到自己的错题记录。
2. 可以按学科、分类、题型、难度、题干关键词筛选，分页结果正确。
3. 可以新增、编辑、删除错题，保存后列表与详情展示一致。
4. 可以上传原图并预览，保存后可再次查看原图。
5. 可以触发 OCR 识别并看到流式结果；当题干为空时可自动回填，否则用户可选择如何处理识别文本。
6. 学科切换后分类选项联动正确，历史数据回显正常。
7. OCR 服务异常、模型未配置、文件上传失败、SSE 中断等情况都有明确错误反馈。
8. 页面实现符合 nquiz 技术基线：Next.js App Router、Tailwind CSS、shadcn/ui、React Hook Form + Zod、TanStack Query。

十、保留/优化/延后建议
1. **保留**：个人错题列表、筛选、增删改、原图关联、OCR 辅助录入。
2. **优化**：页面结构、SSE 状态管理、OCR 回填策略、文件未保存提示、菜单归类与权限语义。
3. **延后**：批量导入导出、结构化 OCR、复习计划、团队共享与全局管理视角。
