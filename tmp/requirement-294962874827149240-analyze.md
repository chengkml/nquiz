结论
- 该需求应按“nquiz 迁移一个完整菜单功能”处理，而不是简单复刻旧页面样式。目标是把 quiz 的 `ScheduleManagement` 从“单文件 Arco 日历页 + Spring Boot calendar 模块”重构为 nquiz 下可持续演进的 Next.js 日程管理模块。
- 核心能力建议保留：个人日程查看（月/周/年）、新增/编辑/删除、状态管理、完成动作、优先级、过期时间、全天标记、个人数据隔离。
- 核心能力建议优化：信息架构、状态流、接口语义、表单校验、查询排序、Todo 联动边界、时间字段一致性。
- 可以重组/延后：农历/节日展示、AI 生成日程、多模型选择、年视图统计增强，建议放到二期，不阻塞一期菜单迁移闭环。

一、目标
- 在 nquiz 中落地 `ScheduleManagement` 菜单，对应 quiz 原路由 `/frame/schedule` 的“日程管理”能力。
- 在保证业务语义不丢失的前提下，按 Next.js App Router + Tailwind + Shadcn UI + React Hook Form + Zod + TanStack Query 的基线重构。
- 形成一个可独立演进的日程领域模块，并为后续 Todo/AI 辅助创建/提醒类能力预留清晰边界。

二、现状
1. nquiz 当前现状
- 当前 nquiz 仍是工程骨架，首页仅为展示页：`src/app/page.tsx`
- 全局仅接入了 QueryProvider：`src/components/providers/query-provider.tsx`、`src/components/providers/app-providers.tsx`、`src/app/layout.tsx`
- 当前没有 Schedule 页面、没有菜单容器、没有日程 domain 的 schema / data access / route handler / form 组件。

- 日程核心实体为 `CalendarEvent`，字段包含：`title`、`descr`、`status`、`priority`、`startTime`、`endTime`、`expireTime`、`allDay`、`completedAt`、`todoId`
- 状态枚举：`SCHEDULED / IN_PROGRESS / COMPLETED / CANCELLED / EXPIRED`
- 优先级枚举：`LOW / MEDIUM / HIGH`

- 日程与 Todo 存在双向关联：
  - 创建日程会同步创建 Todo
  - 创建 Todo 也会同步创建日程
  - 删除/完成/过期会做联动处理

三、代码定位与当前实现
1. 页面与视图实现
- `quiz/frontend/src/pages/Schedule/index.tsx`
- 这是一个“大而全”的单组件，承担了：数据加载、视图切换、月/周/年渲染、模态框表单、AI 生成、SSE 连接、删除确认、完成确认、农历节日展示等全部职责。

2. 前端接口
- `quiz/frontend/src/pages/Schedule/api/index.ts`
- 现有接口：
  - `POST /calendar/search`
  - `GET /calendar/get/:id`
  - `POST /calendar/create`
  - `PUT /calendar/update`
  - `DELETE /calendar/delete/:id`
  - `POST /calendar/:id/complete`
  - `GET /api/calendar/generate/stream?descr=...`

3. 后端领域实现
- `CalendarController.java` 暴露搜索/CRUD/complete/SSE 生成接口。
- `CalendarEventServiceImpl.java` 负责：
  - CRUD
  - 按用户搜索
  - 完成时写入 `completedAt`
  - 创建日程时同步创建 Todo
  - 删除日程时同步删除 Todo
  - 更新日程时仅同步 Todo 的 `expireTime`
  - 流式调用大模型生成日程 DTO
- `TodoServiceImpl.java` 反向做 Todo -> Schedule 的同步创建/删除/完成/过期时间更新。
- `TodoScheduleExpireService.java` 负责按 `expireTime` 扫描 Todo 和 Schedule，把激活状态改为 `EXPIRED`。

四、已识别出的历史包袱 / 不一致点
1. 前端查询参数与后端排序语义不一致
- 前端 `getSchedulesByDateRange` 传了 `sortColumn: 'start_time', sortType: 'asc'`
- 但后端 `CalendarEventServiceImpl.search()` 实际固定按 `e.create_date desc` 排序
- 这说明旧实现存在“前端以为可排序，后端实际忽略”的语义偏差。

2. 查询 DTO 有 `priority`，搜索实现未实际应用 priority 过滤
- `CalendarEventQueryDto` 有 `priority`
- 但 `search()` 中未拼接 priority 条件
- 旧系统存在字段能力与实际查询实现脱节的问题。

3. AI 生成功能的模型参数前后端不一致
- 前端 SSE URL 会传 `modelName`
- 控制器 `streamGenerateEvent(@RequestParam("descr"))` 只接收 `descr`
- 服务端 `llmModelService.getChatModel(null)` 也未使用模型名
- 即前端“可选模型”在旧实现中并没有真正落地。

4. 完成语义存在双轨
- 正确完成路径应走 `/calendar/{id}/complete`，由后端写入 `completedAt`
- 但编辑表单里又允许手动改 `status=COMPLETED`
- `update()` 并不会补写 `completedAt`
- 导致“状态已完成但完成时间为空”的数据风险。

5. Schedule 与 Todo 的同步是强耦合但并不完全对称
- 创建/删除/完成/过期有联动
- 但更新时仅同步 `expireTime`，标题/描述/开始结束时间/优先级/状态并未完整同步
- 当前耦合方式容易产生双向数据漂移。

6. 页面职责过重
- 旧页面是典型“单文件集成所有逻辑”，不适合直接迁移到 nquiz。

五、范围
一期建议纳入范围（必须形成菜单闭环）：
1. 页面定位
- nquiz 新增 ScheduleManagement 菜单页面，建议为 App Router 独立路由，例如 `/schedule` 或归属到业务布局下的对应 route group。

2. 核心查看能力
- 支持月 / 周 / 年三种视图中的至少“月 + 周”完整可用；年视图可保留简化统计版。
- 支持日期导航、跳转今天。
- 支持按当前视图时间范围查询并展示日程。

3. 核心操作能力
- 新增日程
- 编辑日程
- 删除日程
- 标记完成
- 展示状态、优先级、全天、过期时间、完成时间

- 使用 RHF + Zod 管理表单与校验

1. AI 流式生成日程
- 旧系统实现存在参数不一致与强耦合问题，一期不应阻塞主菜单迁移
- 建议先做“手动创建闭环”，再补 AI Assist Create

2. 农历 / 节日展示
- 属于增强型视觉信息，不是日程主链路核心能力

- 一期先保留集成点设计，不强行在前端页面同时做 Todo 联动可视化
- 后端若未来重构为 nquiz 独立服务，也建议先梳理单一事实源

4. 复杂提醒、重复规则、拖拽排期、跨日多段事件
- 旧系统也未看到完整实现，不应在本需求分析中外扩

七、信息架构与视觉设计方向
1. 信息架构建议
- 顶部：页面标题 + 视图切换（月/周/年）+ 日期导航 + 今天 + 新增日程
- 主体：日历视图区域
- 右侧或抽屉：日程详情/编辑表单（建议从旧模态框升级为 Drawer 或双栏）
- 移动端/窄屏：日历区域与表单区域切换式呈现

2. 视觉方向建议
- 不延续 Arco + Less 风格，改用 Shadcn + Tailwind
- 强调信息层级清晰、状态色克制、卡片密度可控
- 月视图优先展示“日数量摘要 + 关键状态徽标”，周视图展示详细卡片
- 完成/过期等状态建议通过 badge + icon + 弱背景区分，不靠大面积高饱和颜色堆砌

3. 样式体系建议
- 页面布局、卡片、徽标、表单全部采用 Tailwind 原子类
- 避免再出现单页面 `style/index.less` 承担全部布局与交互样式的模式

八、实现思路
1. 路由与模块拆分
建议拆成：
- `src/app/(dashboard)/schedule/page.tsx`：页面入口
- `src/features/schedule/components/*`：月视图、周视图、年视图、详情抽屉、表单
- `src/features/schedule/api/*`：查询与 mutation 封装
- `src/features/schedule/schema/*`：Zod schema、DTO 映射
- `src/features/schedule/types/*`：前端领域类型

2. 数据流
- Query Key 建议：
  - `['schedule','calendar', { view, start, end, filters }]`
  - `['schedule','detail', id]`
- Mutation：`createSchedule / updateSchedule / deleteSchedule / completeSchedule`
- 完成、删除、保存后统一 invalidation 当前范围查询
- 避免旧页面那种一个组件里混合本地 state + 手动 reload 的方式

3. 表单边界
- 创建/编辑共用同一 schema
- 必做校验：
  - 标题必填
  - 开始时间必填
  - 结束时间必填
  - `endTime >= startTime`
  - `expireTime` 若存在，不早于 `startTime`（是否强校验可再确认）
- “完成”不通过普通状态下拉直接修改，建议独立操作按钮，统一写 `completedAt`

4. 领域语义建议
- `COMPLETED` 与 `completedAt` 必须保持一致
- `EXPIRED` 建议视为系统计算状态，不开放用户直接编辑
- `allDay` 需明确：是仅 UI 展示，还是改变时间存储规则；旧系统仅存布尔值，一期可先保留为展示/输入能力

5. Todo 联动策略建议
- 一期先保留“存在关联字段”的兼容设计，不要求前端页面直接暴露 Todo 联动入口
- 若后续在 nquiz 自主实现后端，建议明确单一事实源：
  - 方案 A：Schedule 为主，Todo 为派生
  - 方案 B：Todo 为主，Schedule 为派生
  - 不建议继续双向互相创建、互相删除的弱约束模式

6. AI 生成功能建议
- 作为二期独立子能力：`AI 辅助创建日程`
- 前端做成表单中的“AI 填充”入口，不与页面主流程强耦合
- 若保留 SSE，必须补齐模型参数传递、错误协议、可取消流程与解析兜底

九、影响点
1. 前端影响
- 新增 App Router 页面与业务组件
- 新增日历视图组件、表单组件、状态徽标组件
- 新增 query/mutation hooks
- 新增 Zod schema 与 DTO 映射层

2. 后端/接口影响
- 若沿用 quiz 接口，需要增加一个前端 BFF/SDK 层做字段收敛
- 若同步重构后端接口，建议统一：查询排序、过滤、完成动作、状态约束
- 需要明确时区与日期序列化格式，避免 SSR/CSR 解析差异

3. 业务影响
- Schedule 与 Todo 的联动边界需要产品/架构口径统一，否则后续会持续出现状态漂移
- 完成/过期的系统动作与用户动作需重新定义一致性

十、风险 / 疑问
1. 是否必须一期保留“年视图 + 农历/节日”
- 从业务价值看可延后，但若用户强依赖旧体验，需要确认优先级。

2. nquiz 当前是否已有统一菜单/权限框架
- 当前仓库尚未看到现成菜单容器与鉴权边界，Schedule 页面落位需结合后续整体布局方案。

3. 后端来源是否继续复用 quiz 单体接口
- 如果 nquiz 短期复用 quiz API，则需要做兼容层
- 如果 nquiz 会建设新接口，则应借迁移机会修正旧接口语义缺陷

4. Todo 联动是“必须保留”还是“允许降级为后续补齐”
- 从旧代码看联动已存在，但实现并不完全一致
- 需要确认一期验收是否把“联动一致性”作为硬门槛

5. AI 生成是否进入一期验收
- 旧实现本身就有参数未真正生效的问题，不建议把它放进一期硬验收

十一、建议验收标准
一期建议验收标准如下：
1. 路由与页面
- nquiz 中存在可访问的 ScheduleManagement 页面入口
- 页面可在月/周视图中正常查看日程

2. 数据查询
- 页面会按当前视图时间范围拉取日程
- 展示数据仅包含当前用户可见数据
- 查询结果按明确规则排序（建议按 `startTime asc`）

3. CRUD 与完成
- 可新增、编辑、删除日程
- 可单独执行“完成日程”动作，完成后 `status=COMPLETED` 且 `completedAt` 正确写入
- 已过期状态不允许通过普通表单直接手改

4. 表单与校验
- 标题、开始时间、结束时间必填
- 结束时间不能早于开始时间
- 表单错误提示明确

5. 视觉与交互
- 页面使用 Tailwind + Shadcn 风格完成，不依赖 Arco / Less
- 月视图与周视图信息层级清晰，状态可辨识
- 新增/编辑流程在桌面端交互连贯

6. 代码结构
- 页面、视图组件、表单、查询逻辑、schema 分层清晰
- 未出现旧系统那种 1000+ 行单文件承载全部逻辑的情况
- 服务端数据流通过 TanStack Query 管理，而不是散乱 `useEffect` 手搓

十二、保留 / 优化 / 重组 / 延后
1. 保留
- CalendarEvent 核心字段与状态语义
- 月/周/年视图中的日程管理定位
- 新增/编辑/删除/完成核心操作
- 个人数据隔离

2. 优化
- 查询排序与过滤语义
- 完成动作与 completedAt 一致性
- 表单校验与字段边界
- 页面结构拆分与状态管理
- 视觉层级与交互流畅度

3. 重组
- AI 生成从“主流程内强耦合面板”重组为“辅助创建能力”
- Todo 联动从“隐式双向同步”重组为“明确边界的领域集成”

4. 延后
- 农历/节日展示
- 多模型 AI 生成
- 更复杂的提醒/重复规则/拖拽排期

十三、建议实施顺序
- 第一步：先落地页面骨架 + 查询 + 月/周视图
- 第二步：补齐新增/编辑/删除/完成闭环
- 第三步：补齐状态约束、校验、排序与空态/错误态
- 第四步：视需求决定是否接入年视图增强、农历节日、AI 辅助创建、Todo 深度联动
