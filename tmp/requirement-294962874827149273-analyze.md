结论：`quiz` 里的“编排管理页”不是单纯静态入口，而是一个“工作流管理 + 版本化画布编辑 + 手动运行”的轻量编排工作台；但从真实代码看，它目前更像原型级能力，前端画布数据结构与后端执行引擎并没有真正对齐。`nquiz` 不应机械照搬旧页，而应按 Next.js 语境重构为“编排工作台（列表）+ 工作流编辑器（子路由）+ 运行反馈/版本管理”的模块化能力，保留工作流台账、版本发布、画布入口、手动触发这些核心语义，重点修正图结构定义、节点能力、执行反馈和权限边界。

二、真实业务语义与功能边界
1. 这是“编排工作流台账 + 画布入口”，不是完整 BPM 系统
- 工作流可创建、命名、描述、版本化、发布
- 有手动触发执行入口
- 有实例记录能力
- 没有成熟的流程监控、节点级日志、多人协作、审批流等重量级能力

2. 业务对象有三层
- Workflow：业务主对象，承载基本信息与当前发布版本
- WorkflowVersion：每次保存画布形成一个新版本
- WorkflowInstance：一次运行记录

3. 当前产品意图偏 AI Agent / 工作流工作台
- 前端节点命名明显偏 LLM 场景：`llm / knowledge / skill / condition`
- 页面文案也在强调“模型、知识库、技能”编排
- 因此 nquiz 迁移时应默认定位为 AI Orchestration Workbench，而不是通用企业流程引擎

三、旧实现中的关键问题（这是迁移时必须修正的重点）
1. 前后端图结构不一致，旧实现存在原型化痕迹
- 前端保存的是 React Flow 原始节点：节点主要数据在 `node.data`
- 后端执行引擎期望的是 `WorkflowGraph.NodeDef { id, name, type, config }`
- 也就是说前端主要写 `data.label/data.modelName/...`，后端真正读取的是 `name/config`
- 结果是：即使保存了版本，执行时也很可能拿不到后端期望配置，运行语义并不可靠

2. 前端节点库与后端执行器能力不一致
- 前端可拖入：`knowledge / skill / condition`
- 后端 `WorkflowEngine.executeNode()` 真实支持的是：`llm / sql / log / start / end`
- 也就是：前端主推的知识库、技能、条件节点，后端并没有对应执行逻辑；反而后端有 `sql/log`，前端没暴露
- 这说明旧页“可编辑”和“可执行”并未闭环

3. 连线逻辑过于简化
- `WorkflowEngine` 取下一跳时只拿“第一条 source=edge.source 的边”继续执行
- `condition` 字段虽然在 `WorkflowGraph.EdgeDef` 中存在，但没有真正分支求值逻辑
- 所以现阶段并没有完成真正的条件分支编排能力

4. 运行反馈不足
- 列表页点击“运行”后只弹成功消息，并额外查询一次实例列表，但页面并不展示运行结果、输出、步骤日志
- 这导致用户无法在管理页完成“编辑 -> 发布 -> 运行 -> 观察结果”的闭环

5. 权限/数据隔离边界不完全一致
- `search()` 按 `create_user` 过滤，像“我的工作流”
- `searchInstances()` 在未传 `workflowId` 时直接 `findAll()`，存在跨工作流暴露风险
- 画布页元数据通过 `listWorkflows()` 全量拉取后前端再 find 当前项，也不够干净

四、nquiz（Next.js）建议迁移方案
1. 页面定位
建议把“编排管理页”定义为一个模块，而不是单页大杂烩：
- `/orchestration`：编排工作台 / 列表页
- `/orchestration/[workflowId]`：工作流编辑器
- `/orchestration/[workflowId]/runs` 或编辑器内右侧抽屉：运行记录/调试结果

2. 信息架构
A. 列表页保留
- 搜索（关键词）
- 状态筛选
- 工作流卡片/表格
- 新建工作流
- 查看当前发布版本、最近更新时间、运行状态摘要
- 进入编辑器
- 手动运行入口

B. 编辑器页重构为三栏
- 左栏：节点库（只展示当前后端真正支持的节点）
- 中栏：React Flow 画布
- 右栏：节点配置 / 工作流配置 / 版本信息
- 顶部工具栏：保存草稿版本、发布版本、试运行、版本切换

C. 运行反馈区必须补上
- 最近一次运行状态
- 输入参数
- 输出结果 / 错误摘要
- 节点执行轨迹（MVP 至少做到 run summary + error）

3. 数据模型建议（nquiz 语境）
- 按项目约定，新增物理表统一使用 `quiz_` 前缀，不沿用旧表名
- 建议至少拆成：
  - `quiz_orchestration_workflow`
  - `quiz_orchestration_workflow_version`
  - `quiz_orchestration_run`
  - 如要做节点级运行日志，再补 `quiz_orchestration_run_step`
- 工作流版本中的图定义不要直接存 React Flow 原始节点；应存 nquiz 自己定义的标准 schema，例如：
  - `nodes[]: { id, type, name, config, position }`
  - `edges[]: { id, source, target, condition? }`
  - 前端编辑态与后端执行态共用同一份领域结构，再做 UI 层适配

4. 前端实现建议（Next.js）
- App Router 页面组织，画布编辑器作为 client component
- 查询/变更统一走 TanStack Query
- 新建/编辑工作流基础信息使用 React Hook Form + Zod
- 样式用 Tailwind + shadcn/ui，图标用 Lucide
- 版本保存、发布、运行操作都做 mutation，并在成功后精准失效相关 query
- 节点库、节点配置 schema、版本 schema 统一收口到 `src/features/orchestration`

5. 视觉设计方向
- 列表页不必复刻旧 Arco 卡片样式，可改成更像“AI 工作台”的 dashboard：顶部 summary + 筛选条 + 工作流卡片网格
- 编辑器页强调“专业工具感”：深浅对比明确，工具条固定，节点卡片轻量，属性面板分组清晰
- 动画只做微交互：节点面板 hover、保存成功反馈、抽屉/面板切换

五、能力保留 / 优化 / 重组或延后
1. 必须保留
- 工作流列表查询
- 新建/编辑基础信息
- 进入画布编辑
- 版本保存
- 发布当前版本
- 手动运行入口
- 基础运行记录查询

2. 必须优化
- 图结构 schema 统一，消除 `data` vs `config` 断层
- 节点库与执行引擎能力对齐
- 运行结果可见，不再只是 toast
- 权限与数据隔离统一按 workspace/owner 规则处理
- workflow 详情查询改为单条接口，不走全量 list + 前端 find

3. 可以重组
- “编排管理页”只负责管理与入口；“画布编辑”拆到子路由
- 运行记录可以从独立页面改为抽屉/侧栏
- 版本条可从页面中部胶囊列表改为右上角版本下拉 + 历史侧栏

4. 可以延后
- 定时触发 / 事件触发 / API 触发的完整配置化
- 条件分支高级表达式引擎
- 节点级实时日志流
- 多人协同编辑
- 完整节点市场（前提是先把核心执行节点跑通）

六、风险点
1. 如果直接迁移旧前端交互而不重做 graph schema，nquiz 会继承“能画不能稳跑”的问题。
2. 如果节点库继续展示 knowledge/skill/condition，但服务端执行器仍未实现，对用户会形成功能欺骗。
3. 如果运行日志不补齐，编排模块会缺少最关键的可调试性，后续开发成本会持续升高。
4. 旧系统表名是 `orch_*`，nquiz 若要按约定新建表，需明确迁移/兼容策略，避免后续数据切换混乱。

七、未决问题
1. nquiz 首期是否把编排能力定位成“AI 工作流”而非“通用自动化流程”？
2. 首批真正需要上线的节点类型有哪些：只做 `start/llm/knowledge/skill/end`，还是也要带条件分支？
3. 运行结果是否需要节点级 trace，还是首期只展示 run summary / final output 即可？
4. 是否需要保留旧系统中的 `sql/log` 类型；如果保留，前端节点库要不要显式暴露？

八、建议验收标准
1. 在 nquiz 中可创建一个工作流，并进入编辑器完成节点拖拽与配置。
2. 保存版本后，数据库中落的是 nquiz 标准化 graph schema，而不是裸 React Flow 内部结构。
3. 发布指定版本后，列表页和详情页都能正确看到当前发布版本。
4. 手动运行后，可看到 run status、开始/结束时间、最终输出或错误摘要。
5. 前端可见节点与后端可执行节点完全一致，不存在“界面可配但运行无实现”的节点。
6. 列表/详情/运行记录的数据权限边界一致。

九、针对本需求的迁移结论
- 保留：工作流管理、版本化画布、发布、手动运行这四条主链路。
- 优化：统一 graph schema、补齐运行反馈、按 Next.js 模块化拆页、对齐节点与执行器。
- 重组：把“编排管理页”定义成工作台入口，把编辑器与运行反馈拆为子路由/子视图。
- 延后：高级触发器、多人协作、复杂条件引擎、完整节点市场。
