## 目标
将 quiz 的 `DiaryManagement` 迁移为 nquiz 中“当前登录用户的个人日记管理”菜单，保留日记记录、筛选、查看、编辑、删除、归档等核心能力；在 Next.js 语境下按 `App Router + Tailwind CSS + React Hook Form + Zod + TanStack Query + Drizzle ORM` 重构，不照搬旧 Arco 页面壳子。

## 现状
### 代码定位与当前实现
1. 前端页面：`quiz/frontend/src/pages/Diary/index.tsx`
   - 旧路由挂载在 `quiz/frontend/src/router/index.tsx`：`/frame/diary`。
   - 页面本质是单页管理台：顶部筛选 + 表格列表 + 新增/编辑弹窗 + 详情弹窗。
   - 支持字段：标题、正文、日记日期、心情、天气、归档状态。
   - 支持筛选：标题关键字、心情、开始/结束日期、归档状态。
   - 操作闭环：创建、更新、删除、归档/取消归档、查看详情。
2. 前端接口：`quiz/frontend/src/pages/Diary/api/index.ts`
   - `/diary/search`、`/diary/get/{id}`、`/diary/create`、`/diary/update`、`/diary/delete/{id}`、`/diary/{id}/archive`。
3. 后端控制器：`quiz/backend/src/main/java/com/ck/quiz/diary/controller/DiaryController.java`
   - 复用 `BaseController` 提供通用 CRUD，额外增加 archive 接口。
4. 后端服务：`quiz/backend/src/main/java/com/ck/quiz/diary/service/impl/DiaryServiceImpl.java`
   - 搜索 SQL 明确按 `create_user` 过滤，说明这是“个人日记”，不是公共日志。
   - 排序为 `diary_date DESC, create_date DESC`。
   - create 默认补齐 `diaryDate=当天`、`mood=CALM`、`archived=false`。
   - archive / get / update / delete 都依赖创建人权限隔离。
5. 数据模型：`quiz/backend/src/main/java/com/ck/quiz/diary/entity/Diary.java`
   - 表 `diary`，字段为 `title/content/diary_date/mood/weather/archived`，并带 `create_user/create_date/update_user/update_date` 基础审计字段。
   - mood 枚举：`HAPPY/CALM/SAD/ANGRY/TIRED/EXCITED`。
6. nquiz 当前状态：`nquiz/src/app/page.tsx`、`nquiz/src/server/db/schema/example.ts`
   - 当前只有基础首页、React Query Provider、示例 schema，Diary 功能尚未开始迁移，因此本次属于从 0 到 1 的迁移分析，不是局部修订。

## 范围
本需求建议首批纳入：
1. `/diary` 菜单页迁移，承接个人日记管理主入口。
2. 日记列表查询：标题、心情、日期区间、归档状态、分页。
3. 新增/编辑日记表单：标题、正文、日期、心情、天气。
4. 日记详情查看。
5. 删除、归档/取消归档。
6. 当前登录用户的数据隔离与权限校验。
7. Drizzle 新表落库，物理表名遵守 nquiz 约定，建议为 `quiz_diary`。

## 非范围
1. 不做富文本编辑器；旧版正文本质也是纯文本。
2. 不做图片/附件/地理位置/标签/分组扩展；旧页面没有真正暴露这类能力。
3. 不做情绪统计、时间轴、AI 摘要、提醒、日历视图。
4. 不做多人共享、评论、协作日志，这不符合当前个人日记语义。
5. 不强制拆独立详情页，首版可保持同页弹层/抽屉闭环。

## 实现思路
### 信息架构 / 交互
1. App Router 新增 `src/app/diary/page.tsx` 作为菜单页。
2. 页面结构建议为：筛选栏 + 列表区 + 右上角“写日记”按钮。
3. 桌面端优先表格/列表混合布局；移动端退化为卡片列表。
4. 新增/编辑使用 Dialog 或 Sheet；详情优先用 Drawer/Sheet，保留“快速查看不跳页”的旧体验，但交互比旧版弹窗更清晰。
5. 心情用 Badge/Tag 做语义色展示，视觉上延续旧版“情绪可视化”但统一到 Tailwind 设计体系。

### 数据与后端
1. Drizzle 定义 `quiz_diary`：
   - `id`
   - `title`
   - `content`
   - `diary_date`
   - `mood`
   - `weather`
   - `archived`
   - `create_user/create_date/update_user/update_date`
2. 索引建议：
   - `(create_user, diary_date desc)`
   - `(create_user, archived)`
   - `(create_user, mood)`
3. API 建议保留旧语义即可：list/search、detail、create、update、delete、archive；这样迁移认知成本最低。
4. 服务端必须继续按当前用户过滤，任何详情/更新/删除/归档都必须校验数据归属，避免越权。

### 前端工程方案
1. 查询与变更统一走 TanStack Query。
2. 表单统一走 RHF + Zod，默认值与旧逻辑一致：`diaryDate=今天`、`mood=CALM`、`archived=false`。
3. 日期字段建议在前后端之间统一为 `YYYY-MM-DD`，避免时区导致跨天。
4. 筛选条件与分页建议 URL Search Params 化，便于刷新恢复和后续分享定位。

## 影响点
1. 路由：新增 nquiz 日记菜单路由与导航入口。
2. 数据库：新增 `quiz_diary` 表及索引，不应直接沿用旧表名 `diary`。
3. 服务端：新增 Diary 模块的 route handler / server action / service 层封装。
4. 前端：新增 diary 页面、筛选组件、表单组件、详情抽屉、query keys。
5. 权限：依赖现有登录态获取当前用户，否则无法保持“个人日记”语义。

## 风险 / 疑问
1. **登录态来源未定**：nquiz 当前仍是骨架，需确认用户身份在 Next.js 中如何注入服务端查询。
2. **日期时区风险**：旧版是 `LocalDate`，nquiz 若直接用 `Date`/ISO 时间戳，容易出现跨天问题。
3. **分页与筛选协议未统一**：旧后端是 `pageNum/pageSize` 且 `pageNum` 从 0 开始，nquiz 需明确是否保持兼容还是改造后整体统一。
4. **详情形态待定**：首版若改成 Drawer 而非 Modal，属于交互优化，但需确认是否接受与旧版视觉差异。
5. **归档语义要保留**：旧系统把 archived 当成状态切换而非软删除，nquiz 不应误收敛为“删除即归档”。

## 建议验收标准
1. 进入 `/diary` 后可看到仅属于当前登录用户的日记列表，默认按日记日期倒序。
2. 可按标题、心情、日期区间、归档状态筛选，并支持分页切换。
3. 可新增日记；标题、正文、日期、心情必填/必选校验生效；默认值符合旧逻辑。
4. 可编辑本人日记并成功回显更新结果。
5. 可查看日记详情，正文按换行保留展示。
6. 可删除本人日记，删除后列表正确刷新。
7. 可归档与取消归档，状态展示与筛选结果正确联动。
8. 非本人数据无法被查询详情、修改、删除、归档。
9. 数据库表、索引、字段命名符合 nquiz 新表 `quiz_` 前缀约定。

## 保留 / 优化 / 延后建议
1. **保留**：个人日记语义、核心 CRUD、筛选、归档、心情枚举、天气字段。
2. **优化**：由旧版 Arco + 手写状态切到 Next.js + Query + RHF + Zod；筛选状态 URL 化；详情弹层体验优化；数据库命名统一为 `quiz_` 前缀。
3. **延后**：情绪统计、时间轴、附件、多端草稿、AI 总结、日历视图等增强能力。
