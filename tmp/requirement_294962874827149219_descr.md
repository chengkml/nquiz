## 目标

将 `quiz` 中的 `ExamHistoryManager` 迁移为 `nquiz`（Next.js App Router）里的“我的考试历史 / 答卷复盘”能力，保留**当前登录用户查看自己历史答卷、进入详情复盘、删除自己的答卷、提交后快速查看结果**这些核心业务语义；不要求照搬旧页面布局，但要把旧实现里路由混乱、伪筛选、权限不完整等历史包袱在新版本里收敛掉。

## 现状

### 1. `quiz` 里的真实代码位置与行为

前端核心文件：
1. `quiz/frontend/src/pages/Exam/History/index.tsx`
2. `quiz/frontend/src/pages/Exam/History/Detail/index.tsx`
3. `quiz/frontend/src/pages/Exam/History/api/index.ts`
4. `quiz/frontend/src/pages/Exam/Take/index.tsx`
5. `quiz/frontend/src/router/index.tsx`

后端核心文件：
1. `quiz/backend/src/main/java/com/ck/quiz/exam/controller/ExamController.java`
2. `quiz/backend/src/main/java/com/ck/quiz/exam/service/impl/ExamServiceImpl.java`
3. `quiz/backend/src/main/java/com/ck/quiz/exam/dto/ExamResultDetailDto.java`
4. `quiz/backend/src/main/java/com/ck/quiz/exam/dto/ExamResultHistoryItemDto.java`
5. `quiz/backend/src/main/java/com/ck/quiz/category/controller/CategoryController.java`

已确认的旧系统真实能力：
- `/frame/history` 是**当前登录用户自己的考试历史列表页**，不是管理员看所有人答卷。
- 列表展示：`examName`、`totalScore`、`userScore`、`correctCount`、`submitTime`，行点击进入详情。
- 支持删除答卷，且后端删除接口确实校验 `examResult.userId == authentication.getName()`。
- 详情页展示整份答卷、题型、用户答案、标准答案、解析、知识点，并支持“只看错题”和右侧题号导航。
- 提交考试后，`Take/index.tsx` 会把 `lastSubmittedResultId` 写入 `sessionStorage`，再跳转 `/frame/history`，历史页首次加载时自动打开对应详情。

### 2. 旧实现里的历史包袱 / 不一致点

#### 2.1 前端筛选 UI 与后端能力不一致
- 历史页前端传了 `examName`、`subjectId`，还展示了学科/分类树。
- 但后端 `ExamController.listUserResults(...)` 只接收 `pageNum/pageSize`，`ExamServiceImpl.listUserResults(...)` 也只按 `userId` 查询并按提交时间倒序分页。
- 结论：旧页面里的“名称搜索”“左侧学科树筛选”在现状下基本属于**伪筛选 / 未真正落库的交互**，不能作为 nquiz 必须 1:1 保留的强业务约束。

#### 2.2 路由定义重复且语义混乱
- `quiz/frontend/src/router/index.tsx` 同时把 `history` 放进菜单路由，又在 `/frame` children 里再写了一次 `{ path: "history", element: <ExamHistoryPage /> }`。
- 详情页使用 `/frame/history/result/:id`，列表仍挂在菜单体系下。
- 这说明旧路由组织存在重复和职责不清，nquiz 不应照搬。

#### 2.3 答卷详情接口缺少“仅本人可看”校验
- `deleteExamResult(...)` 校验了 `userId`。
- 但 `getExamResultById(resultId)` 直接按 `resultId` 查询后返回，没有再次校验当前登录用户是否为答卷拥有者。
- 这在 nquiz 里应视为需要修复的安全缺口，而不是保留旧行为。

### 3. `nquiz` 当前现状

`nquiz` 目前仍是骨架阶段，已确认只有基础 Next.js 结构和 Query Provider：
- `src/app/page.tsx`
- `src/app/layout.tsx`
- `src/components/providers/query-provider.tsx`
- `package.json`

当前仓库里**尚无 Exam / History 领域页面、数据访问层、Drizzle schema、服务端路由或 UI 组件实现**。因此本需求本质上不是“微调一个已有页面”，而是**按菜单粒度从零规划并落地考试历史模块**。

## 页面定位

该模块在 nquiz 中应定义为：
- 面向普通登录用户的“我的考试历史”入口
- 承接考试提交后的结果复盘
- 提供历史答卷检索、复盘、删除本人记录的能力
- 是“考试作答链路”的后置页面，不是考试管理后台

## 功能边界

### 范围内（本次迁移建议覆盖）
1. 我的历史答卷列表
2. 历史答卷详情 / 复盘页
3. 只看错题
4. 题号导航
5. 删除本人答卷
6. 从考试提交完成后进入历史或详情页
7. 基础筛选（至少关键字；若数据模型准备好，再加真实 subject/category 过滤）
8. 登录态与数据权限控制

### 非范围（本次不建议绑进首批迁移）
1. 试卷管理、组卷、发布考试
2. 考试作答引擎与评分逻辑重写
3. 全局成绩分析大盘 / 管理员看所有人历史
4. 导出 PDF / Excel、批量删除、批量归档
5. 复杂学习统计（按周趋势、能力雷达、错因聚类）

## 信息架构与页面组织建议

建议在 nquiz 中拆成两个语义明确的路由：
- `/exam/history`：历史列表页
- `/exam/history/[resultId]`：答卷详情页

而不是继续沿用 `/frame/history/result/:id` 这种强耦合旧布局路由。

### 列表页建议结构
1. 顶部页头：标题、副标题、必要说明
2. 筛选区：关键字、学科（可选）、时间范围（可选）
3. 主体区：答卷列表（桌面端表格 / 移动端卡片）
4. 行级操作：查看详情、删除
5. 空状态 / 错误状态 / 加载状态

### 详情页建议结构
1. 顶部摘要：试卷名称、提交时间、得分、正确题数
2. 操作区：返回列表、只看错题切换
3. 主内容区：题目与作答内容
4. 右侧或浮动导航：题号导航、当前题高亮
5. 答案解析区：正确答案、解析、知识点

## 视觉设计方向

- 不再沿用 Arco + 老式管理页拼装感。
- 使用 `shadcn/ui + Tailwind CSS` 统一信息密度和间距体系。
- 列表页优先强调“结果回顾”和“快速定位最近答卷”，不是堆满后台管理操作。
- 详情页优先阅读体验：题目卡片化、对错状态可视化、答案/解析/知识点分层展示。
- 错题态、当前导航态、得分等级态分别做明确颜色语义，但避免旧页中过度依赖彩色 Tag。

## 样式体系建议

- 页面容器、筛选栏、列表、题卡都用 Tailwind 贴近组件实现。
- 基础组件优先复用 shadcn/ui 的 `Card`、`Button`、`Table`、`Badge`、`Checkbox`、`Tabs`、`ScrollArea`、`AlertDialog`。
- 交互反馈（切换错题、滚动定位、详情切换）可少量引入 `Framer Motion` 做过渡，但不要做噪音动画。

## 交互方式建议

### 保留的核心交互
1. 点击列表项进入答卷详情
2. 详情页只看错题
3. 题号导航快速定位
4. 删除本人答卷
5. 提交考试后快速看到结果

### 建议优化的交互
1. **提交后跳转方式**：
   - 旧实现依赖 `sessionStorage.lastSubmittedResultId`，是脆弱的页面级技巧。
   - nquiz 建议改为：提交成功后直接跳转 `/exam/history/[resultId]?from=submit`，或跳转 `/exam/history?focusResultId=xxx` 并用 URL 驱动状态。
2. **筛选状态 URL 化**：
   - 关键字、学科、分页建议落到 URL search params，便于刷新恢复和分享。
3. **删除确认**：
   - 用更明确的 destructive dialog，提示“仅删除本人历史记录，不影响试卷本身”。
4. **详情导航**：
   - 右侧固定导航在桌面端保留；小屏改成底部抽屉或浮动按钮，不必死搬旧布局。

## 数据结构与业务语义

从旧代码可确认的结果域核心字段：
- 列表项：`resultId`、`examId`、`examName`、`totalScore`、`userScore`、`correctCount`、`submitTime`
- 详情：`resultId`、`examId`、`userId`、`totalScore`、`correctCount`、`submitTime`、`answers[]`
- 答题明细：`examQuestionId`、`correct`、`score/gainScore`、`userAnswers[]`、`standardAnswers[]`、`question`
- 题目结构：`id`、`type`、`content`、`options`、`answer`、`explanation`、`knowledgePoints/knowledges`

业务语义上，这个模块管理的是**考试结果快照**，不是题目主数据本体；删除历史答卷意味着删除结果记录与结果答案，不应影响试卷定义与题库题目。

## 数据流 / 状态管理建议

### 前端
- 列表、详情走 `TanStack Query`
- 筛选表单走 `React Hook Form + Zod`
- URL search params 承载分页 / 搜索 / subject 等条件
- “只看错题”这种纯视图状态可留在本地组件状态

### 服务端 / BFF
建议在 nquiz 中提供稳定的 typed contract，而不是让页面直接依赖旧接口偶然行为：
- `GET /api/exam/results`
- `GET /api/exam/results/[resultId]`
- `DELETE /api/exam/results/[resultId]`

如果考试域后端尚未完全迁移到 nquiz，可分两阶段：
1. **阶段一**：Next.js BFF / route handler 代理旧 quiz 服务，但在 BFF 层统一 DTO、补权限与筛选语义。
2. **阶段二**：考试域数据逐步迁入 nquiz 自己的服务与数据库层。

## 实现思路

### 方案建议：按“最小闭环”分两期

#### 一期（建议首批必做）
目标：先把“我的历史 + 详情复盘”闭环跑通。

包含：
1. `/exam/history` 列表页
2. `/exam/history/[resultId]` 详情页
3. 当前用户本人数据权限校验
4. 删除本人答卷
5. 提交后直接跳详情或带参数跳列表
6. 真实可用的关键字搜索（如果后端暂不支持，就不要先放假搜索框）

#### 二期（按数据准备情况补）
1. 学科 / 分类真实过滤
2. 最近一次考试结果高亮
3. 错题汇总入口联动
4. 时间范围、排序方式、更多统计摘要

### 为什么这样拆
- 旧页里最核心的价值是“答卷回看 + 错题复盘”，不是左树筛选。
- 旧树筛选在现状代码里并未真正由后端支持，属于可重组能力。
- nquiz 当前尚无 exam 域代码，应优先交付真正可用的个人历史闭环，而不是把伪功能完整复制过去。

## 影响点

### 前端影响
1. 新增 App Router 页面与路由层级
2. 新增考试历史 DTO / query key / API client
3. 新增列表与详情组件、题目渲染组件
4. 调整考试提交成功后的跳转逻辑

### 服务端影响
1. 增补或重构答卷列表查询接口，使筛选语义与前端一致
2. 详情接口补“仅本人可看”校验
3. 删除接口继续保留“仅本人可删”约束
4. 若支持 subject/category 过滤，需要明确结果与学科/分类的映射来源

### 数据层影响
如果 exam 域数据也在 nquiz 侧重建，建议沿用结果快照语义，相关新物理表继续遵守 `quiz_` 前缀约定，例如：
- `quiz_exam_result`
- `quiz_exam_result_answer`

若一期仍复用旧服务，可先不急于在当前仓库落新表，但接口 contract 要按 nquiz 标准先收口。

## 风险 / 疑问

1. **筛选语义未定**
   - 旧页面展示了 subject/category 树，但后端没真正支持。
   - 需要产品上明确：nquiz 首期是否真的要求按学科/分类过滤历史答卷。

2. **分类过滤的技术映射可能不简单**
   - 历史答卷本身未直接展示 `subjectId/categoryId` 字段。
   - 如果要支持分类过滤，可能要通过 `exam -> examQuestion -> question/knowledge -> category/subject` 做关联，需提前确认性能和口径。

3. **旧详情接口存在越权风险**
   - nquiz 必须修复，不能照搬。

4. **nquiz 当前无 exam 域骨架**
   - 这意味着开发时要一并定义 DTO、页面、查询层、权限边界，而不是只迁一个 JSX 页面。

5. **题型渲染需要统一抽象**
   - 旧详情页在一个页面里内联处理 SINGLE / MULTIPLE / BLANK / SHORT_ANSWER。
   - nquiz 若不做题型渲染抽象，后续 Exam Take / Exam Detail / History Detail 之间容易重复代码。

## 建议验收标准

1. 登录用户进入 `/exam/history`，能看到**自己的**历史答卷分页列表。
2. 列表至少正确展示：试卷名、试卷总分、用户得分、正确题数、提交时间。
3. 点击任一记录可进入 `/exam/history/[resultId]` 查看完整答卷详情。
4. 详情页支持“只看错题”，并可通过题号导航跳转到对应题目。
5. 详情页可正确展示用户答案、标准答案、解析、知识点（有数据时）。
6. 删除答卷后，该记录从列表消失，且不会删除试卷定义或题库题目。
7. 非答卷本人访问他人 `resultId` 时会被拒绝（接口和页面都成立）。
8. 提交考试成功后，用户能稳定到达对应结果页或历史页，不依赖 `sessionStorage` 黑魔法。
9. 若首期提供筛选能力，则筛选必须是**真实生效**的，不能保留旧系统那种前端有 UI、后端无语义的假筛选。

## 结论（保留 / 优化 / 延后）

### 建议保留
- 我的历史答卷列表
- 历史详情复盘
- 只看错题
- 题号导航
- 删除本人记录
- 提交后快速看结果

### 建议优化
- 路由重组为 App Router 语义路径
- 用 Query + URL 状态替代页面内散乱状态
- 用 shadcn/ui + Tailwind 重做信息层级
- 修复详情越权
- 把伪筛选改成真实筛选，或首期先不暴露

### 建议可重组 / 延后
- 左侧学科/分类树完整迁移
- 高级筛选与统计看板
- 导出、批量操作、归档等扩展能力
