结论：`TokenUsagePage` 在 quiz 中不是“完整的 Token 审计中心”，而是一个面向当前登录用户的 **Token 用量自助统计页**。旧页核心能力是：按模型、按业务类型、按日期查看自己的 token 消耗、请求次数与成本汇总；它挂在 `/frame/token-usage` 菜单下，属于运营/资源消耗观察面板，而不是管理员级的全局账单系统。迁移到 nquiz 时应保留“个人维度 Token 使用概览 + 多维统计趋势”主语义，但不建议照搬旧版 Arco Tab+Table 页面；更适合重构为 Next.js App Router 下的“个人 AI 使用洞察页”，强调概览卡片、趋势图、维度切换和清晰筛选，同时补上旧实现中的数据边界与可扩展性问题。

二、旧系统真实功能与业务语义
1. 页面默认展示当前登录用户自己的 token 用量，而不是全员统计。证据：前端只调用 `GET /token-usage/my-statistics/by-model`、`/by-business`、`/by-date`；后端这些接口都会从 `Authentication` 注入 `userId`。
2. 页面主视图分三块：
   - 按模型统计（model）
   - 按业务类型统计（business）
   - 按日期统计（date，可附带日期范围 + 模型筛选）
3. 顶部四个概览卡片来自 `modelStats` 汇总，而不是后端单独返回 overview：
   - 总 Token
   - 输入 Token
   - 输出 Token
   - 总成本
4. 统计行结构统一为 `TokenUsageStatDto`：`dimension` + `promptTokens` + `completionTokens` + `totalTokens` + `totalCost` + `requestCount`。这说明旧页本质上是“统一统计维度下的汇总表”，并不强调原始明细。
5. 业务类型是枚举字符串映射后展示中文，当前前端内置映射：`CHAT/QUESTION/OCR/KNOWLEDGE/DATASOURCE/FUNCDOC/MINDMAP/MERMAID/CALENDAR`。因此该模块语义不仅覆盖聊天，也试图成为多个 AI/内容生成功能的公共资源计量入口。
6. 按日期统计支持两个过滤条件：日期范围、模型名；模型下拉选项直接来自当前“按模型统计”的维度列表。
7. 旧页没有展示明细记录、会话详情、异常记录、原始请求/响应内容，虽然前后端 DTO 和接口已经为这些能力留了口子（`queryRecords`、`queryBySessionId`、`TokenUsageRecordDto`）。这意味着旧页是“轻分析页”，不是“审计追踪页”。
8. 成本来自后端根据 `LLMModel.inputPricePer1k` / `outputPricePer1k` 动态计算，不是前端估算；因此页面上的金额具有“配置驱动计费”语义。

三、从真实代码看出的当前实现问题与历史包袱
1. **前后端边界不一致**：前端 API 文件定义了 `queryStatistics`、`queryRecords`、`queryBySessionId`，但页面实际只消费了 `my-statistics/*` 三个接口，存在“接口已扩出、页面没承接”的半成品状态。
2. **`queryRecords` 实现不正确/不完整**：`TokenUsageServiceImpl.queryRecords()` 直接 `findAll()` 后映射 DTO，完全没使用 `TokenUsageQueryDto` 的日期、模型、业务类型、用户过滤条件；如果后续页面接入明细列表，这会直接产生越权与性能问题。
3. **`queryBySessionId` 缺少显式权限校验**：控制器直接按 `sessionId` 查记录返回，没有基于当前用户做所有权校验；旧页面没用上它，但一旦开放会话明细，就有潜在越权风险。
4. **用户归属写入逻辑存在歧义**：`recordUsage(..., userId)` 方法签名接收 `userId`，但当前实现没有显式 `setCreateUser(userId)`，实际依赖 JPA 审计上下文自动写入。若未来在异步任务、批处理、工作流等非标准请求线程中记账，可能出现归属缺失或归属错误。
5. **成本精度和货币语义比较粗糙**：前端直接把 `totalCost` 格式化成 `¥`；但模型价格配置未必一定是人民币，也没有汇率、计费版本、生效时间等概念，更像“粗略成本估算”。
6. **业务类型中文映射硬编码在前端**：一旦新增业务类型，前端不更新就会直接显示原始枚举值，扩展性一般。
7. **页面形态偏表格化**：全部是统计卡 + Table，缺少趋势图、异常波动提醒、占比可视化，不利于用户快速感知“最近用量是否异常”。
8. **页面只统计“我的数据”**：这本身没问题，但名称 `TokenUsagePage` 比较泛，容易让人误解为系统级计量中心。nquiz 中建议明确命名语义，例如“我的 AI 用量”或“AI Token 使用情况”。
9. **旧样式和交互较基础**：`index.less` 只做了简单 padding / header / 表格居中，没有为复杂筛选、趋势、空状态、错误状态做更细设计。
10. **打点覆盖面待确认**：从当前代码证据看，至少聊天场景已接入；但 QUESTION/OCR/KNOWLEDGE 等业务类型虽然出现在枚举注释与前端映射里，本次未继续追到所有调用点，说明这些类型可能是“规划中的统一枚举”或“部分已接入、部分待接入”。这在迁移时要明确标注为待确认范围，而不是假设全部已稳定落地。

四、nquiz 迁移目标
1. 保留旧系统的核心语义：让当前用户清楚知道自己在不同模型、不同业务、不同日期维度上的 AI 资源消耗情况。
2. 在 nquiz 中把它定义为 **个人 AI 使用洞察页**，而不是管理员账单后台。
3. 迁移后重点不是 1:1 复刻旧表格，而是建立一个可持续扩展的数据观察面板，为后续接入更多 AI 功能模块留统一计量入口。

五、建议范围（本次迁移应覆盖）
1. 页面入口与路由：nquiz 中提供稳定菜单入口，例如 `app/(dashboard)/insights/token-usage/page.tsx` 或 `app/(dashboard)/ai/token-usage/page.tsx`。
2. 顶部概览：总 token、输入 token、输出 token、估算成本、请求次数。
3. 维度切换：至少保留“按模型 / 按业务类型 / 按日期”三种维度。
4. 日期筛选：支持起止日期范围。
5. 模型筛选：在日期维度下支持按模型过滤。
6. 数据加载与刷新：支持首次加载、主动刷新、加载态、空态、失败态。
7. 中文展示：业务类型、字段说明、成本说明需要更清晰的人类可读表达。
8. 数据边界：默认仍只看“当前用户自己”的数据。

六、建议非范围（本次迁移先不做，或明确延后）
1. 系统管理员全局账单中心。
2. 原始请求/响应明细审计页。
3. 按会话 drill-down 的详细时间线查看器。
4. 成本分摊、预算阈值、告警通知。
5. 多币种/汇率/计费版本治理。
6. 财务结算级精确账单导出。
7. 所有业务类型的历史追补与补录修数工具。

七、实现思路（nquiz / Next.js 重构版）
1. 路由与页面组织：
   - 使用 App Router 放在 dashboard 域。
   - 页面组件拆成：`overview-cards`、`dimension-tabs`、`date-filter-bar`、`stats-table`、`usage-empty-state` 等。
2. 数据获取：
   - 用 TanStack Query 建立 `tokenUsage.myOverview` / `tokenUsage.byModel` / `tokenUsage.byBusiness` / `tokenUsage.byDate(filters)` 查询。
   - 若后端仍沿用旧接口，可在 nquiz 侧封装统一 typed fetcher；不要把旧 axios API 文件结构直接照搬进页面组件。
3. 交互设计：
   - 默认先加载 overview + byModel + byBusiness；按日期维度在用户进入该 tab 或变更筛选时再查询。
   - 支持手动刷新当前视图。
   - 重置筛选时恢复默认近 7/30 天策略，或回到“不过滤”的全量个人统计（需产品确认）。
4. 展示层建议：
   - 概览卡片用 shadcn + Tailwind 重构，突出趋势和语义；
   - “按日期”优先考虑折线/柱状图 + 表格双视图，而不是只有表格；
   - “按业务类型”可增加占比条/饼图，帮助快速发现大头消耗来源。
5. 数据模型建议：
   - 前端显式区分 `UsageStatItem` 与 `UsageFilter`；
   - 后端若重建表，物理表应遵循 nquiz 约定，使用 `quiz_` 前缀，如 `quiz_token_usage`。
6. 服务端建议：
   - 保留按模型、按业务、按日期的聚合接口；
   - 若需要扩展 overview，可新增专门 overview 接口，避免前端再从 modelStats 二次聚合；
   - 修复 `queryRecords` 的过滤与权限问题，即便本次页面不接明细，也应作为后续能力预埋时的前置修正。
7. 权限与安全：
   - 默认用户仅能看自己的数据；
   - 如果以后扩展管理员视图，必须单独 API 和权限模型，不要复用当前接口悄悄放开 `userId` 查询。
8. 文案与命名：
   - 页面标题建议从“Token 使用统计”优化为“我的 AI 用量”或“AI Token 用量”，减少管理员报表误解。
   - 金额字段建议标注为“估算成本”，避免被理解为财务结算值。

八、影响点
1. 前端页面层：新增 nquiz 统计页、筛选组件、表格/图表组件、query hooks。
2. 前端全局导航：需要把该页面接入 nquiz 新菜单体系。
3. 数据访问层：需要统一 token usage API 封装与 query key。
4. 后端统计接口：若直接复用 quiz 旧接口，至少要验证认证、返回结构和跨域/网关接入方式；若重建，则需要同步实现聚合查询。
5. 计量打点链路：未来若要让该页真正有价值，还需要更多 AI 业务在调用模型后统一接入 token usage 记录。
6. 模型配置域：成本计算依赖模型价格配置，因此模型管理和 token 统计存在间接耦合。

九、风险 / 疑问
1. **业务覆盖风险**：当前只明确验证到 CHAT 已接入打点，其它业务类型是否真实入表、入表比例如何，需要在开发前补核。
2. **数据准确性风险**：成本是基于当前模型单价动态算出的估算值；如果模型单价后改，历史成本口径是否追溯一致需要确认。
3. **权限风险**：`queryBySessionId` / `queryRecords` 旧实现存在潜在越权和全量扫描问题，nquiz 若扩展明细能力不能直接照搬。
4. **归属风险**：当前 user 归属依赖审计上下文，异步/批处理场景记账是否可靠需确认。
5. **命名风险**：页面名过泛，可能与未来“管理员全局用量中心”冲突，建议在 nquiz 里提前做命名区分。
6. **nquiz 基础设施风险**：当前项目还没有完整 dashboard、认证壳、业务 API 层，TokenUsagePage 迁移不是单页面平移，而是要落到新的系统骨架里。

十、建议验收标准
1. 登录用户可以在 nquiz 中进入“我的 AI 用量 / Token 用量”页面，并成功看到自己的统计数据。
2. 页面至少提供总 token、输入 token、输出 token、请求次数、估算成本 5 类核心概览。
3. 支持按模型、按业务类型、按日期三种维度查看统计结果。
4. 按日期视图支持日期范围筛选，且可选模型筛选。
5. 页面具备明确的加载态、空态、错误态和手动刷新能力。
6. UI 符合 nquiz 技术栈与风格：Next.js App Router + Tailwind CSS + shadcn/ui + TanStack Query。
7. 默认数据边界为“仅当前用户可见”；不得意外暴露其他用户 token 使用记录。
8. 页面文案明确说明金额为估算成本，不承诺财务结算精度。
9. 若预埋明细接口，则其查询必须具备有效过滤和权限校验，不能延续 `findAll()` 级别实现。
