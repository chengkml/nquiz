## 1. 项目目标

`nquiz` 不是给 `quiz` 打补丁，而是承担 **“将 quiz 重构成 Next.js 版本”** 的工作。

默认目标：

- 保留原有业务能力与核心流程
- 用现代前端工程方案重构 UI、状态管理、表单和数据层
- 形成结构清晰、类型安全、便于 AI 持续开发的新代码库

## 2. 技术栈基线（默认执行标准）

### 2.1 UI 与交互层

- 样式：`Tailwind CSS`
- 组件：`Shadcn UI`
- 图标：`Lucide React`
- 动画：`Framer Motion`

### 2.2 数据与工程化

- 框架：`Next.js`（优先 App Router）
- ORM：默认优先 `Drizzle ORM`
- 备选 ORM：`Prisma`（仅当项目基线已确定或程凯明确指定时）
- 表单：`React Hook Form + Zod`
- 服务端状态：`TanStack Query`
- **数据库命名约定**：`nquiz` 新增数据库物理表名统一使用 `quiz_` 前缀，避免与原工程已有表重名

## 3. 迁移总准则（程凯指定，必须遵守）

### 3.1 迁移粒度

- 默认按 **功能菜单粒度** 进行迁移。
- 不按零散页面碎片迁移，不按纯技术层拆解迁移。
- 每次迁移应尽量围绕一个完整菜单功能形成闭环，包括：页面入口、核心交互、依赖接口、必要状态、必要表单与数据展示。

### 3.2 功能覆盖要求

- 迁移后必须以 **覆盖 `quiz` 原有功能** 为目标。
- 不允许借“重构”为由偷偷删减关键能力、权限约束、核心交互或主要业务分支。
- 允许重写实现方式，但不允许丢失业务能力。

### 3.3 技术实现要求

- 实现方式、样式、交互风格，不要求复刻旧 `quiz` 的写法。
- 默认遵循当前 `Next.js` 技术栈的最佳实践即可。
- 可以重构页面结构、组件拆分、数据获取方式、表单方案和样式体系，但要确保业务语义仍然一致。

### 3.4 差异汇报要求

- 每迁移完一个功能，必须主动汇报：
  1. 已迁移范围
  2. 与原 `quiz` 功能相比的差异
  3. 已覆盖能力
  4. 尚未覆盖能力
  5. 风险与后续建议
- 如果存在有意调整，也必须明确说明“为什么这样调整”。

## 4. 默认架构偏好

- 页面与布局优先采用 Next.js App Router 组织。
- UI 组件优先在 Shadcn UI 基础上定制，而不是重复造轮子。
- 样式优先写在组件附近，用 Tailwind 表达，不把样式逻辑四处散落。
- 表单默认使用 React Hook Form 管理状态，Zod 统一校验与类型边界。
- 数据请求默认使用 TanStack Query，避免写大量手搓请求状态和混乱副作用。
- 图标默认使用 Lucide React。
- 动画只在需要表达反馈、切换、层次时引入 Framer Motion。

## 5. 重构工作原则

- **业务优先**：先保证业务链路和关键页面行为正确，再追求结构完美。
- **逐步迁移**：按模块/页面/链路分批重构，不做无边界大爆炸迁移。
- **最小闭环**：每次改动至少形成一个可验证闭环。
- **类型安全**：新代码优先保证类型、Schema、接口边界清晰。
- **保留语义**：旧系统的业务语义、字段含义、权限约束不能在重构中被偷偷改变。
- **允许重设计实现，不允许丢失业务能力**。

## 6. 已安装技能与触发规则

### 6.1 `shadcn-ui`
触发条件：
- 新页面搭建
- 组件抽象
- 对话框、表格、表单容器、导航、弹层、命令面板等 UI 结构开发

执行要求：
- 优先复用组件源码，不写重复基础组件
- 允许按项目风格调整组件实现，但保持可维护性

### 6.2 `tailwind-css-patterns`
触发条件：
- 页面布局
- 视觉层级调整
- 响应式适配
- 组件局部样式整理

执行要求：
- 样式尽量与组件同文件表达
- 避免回退到分散且难维护的传统 CSS 堆砌

### 6.3 `framer-motion-animator`
触发条件：
- 页面切换反馈
- 弹层/抽屉/卡片/列表进入退出动画
- 用户交互引导与状态变化过渡

执行要求：
- 动画服务于信息表达，不做噪音动画
- 注意性能与时长控制

### 6.4 `drizzle-orm-patterns`
触发条件：
- 设计数据库 Schema
- 编写查询/写入逻辑
- 建立类型安全的数据访问层

执行要求：
- 默认按 Drizzle ORM 方案落地
- 若后续项目明确切换 Prisma，再整体调整，不混搭双 ORM

### 6.5 `react-hook-form-zod`
触发条件：
- 新增或重构表单页面
- 输入校验
- 字段联动、默认值、错误提示、提交前校验

执行要求：
- 表单默认使用 RHF + Zod
- Schema 是字段约束与类型边界的唯一可信来源之一

### 6.6 `tanstack-query`
触发条件：
- 列表查询
- 详情查询
- CRUD 后的缓存同步
- 乐观更新、失效重取、分页、筛选

执行要求：
- 服务端数据优先走 Query / Mutation 模式
- 避免直接把请求写进页面 `useEffect` 里

## 7. 输出要求

3. 风险与影响
4. 验证结果或验证命令
5. 与原 `quiz` 功能差异
6. 下一步建议（可选）

## 8. 风险控制

- 高风险操作前必须先确认。
- 不能在未验证情况下说“已完成/已解决”。
- 不泄露任何敏感信息、token、密钥、cookie、凭据。
- 若某项无法验证，必须明确标注未验证项。
